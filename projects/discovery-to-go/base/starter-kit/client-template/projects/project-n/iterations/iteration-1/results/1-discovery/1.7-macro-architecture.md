# Macro Architecture — FinTrack Pro

> **Fase:** 1 — Discovery | **Iteracao:** 1 | **Bloco:** 1.7 | **Status:** Rascunho

---

## Decisao de Arquitetura

| Aspecto | Decisao |
|---------|---------|
| Estilo | **Monolito modular** |
| Justificativa | Equipe de 5 pessoas — microservicos gerariam overhead operacional desproporcional |
| Quantidade de modulos | 4 dominios internos com fronteiras bem definidas |
| Evolucao | Migracao para microservicos prevista na Fase 2, condicionada a necessidade de escala |
| Validacao | CTO concorda que monolito modular e a abordagem correta para o tamanho da equipe |

O monolito modular permite manter a simplicidade de deploy e depuracao de um monolito enquanto preserva separacao logica entre dominios. Cada modulo tem seu proprio namespace, repositorio de dados logico e interface publica interna.

---

## Decomposicao em Modulos

### Modulo 1 — Ingestao (Ingestion)

| Atributo | Detalhe |
|----------|---------|
| Responsabilidade | Conexao com Open Finance (Pluggy), download de extratos, normalizacao de dados |
| Processamento | **Assincrono** via worker BullMQ (Redis) |
| Frequencia | A cada 4 horas no MVP |
| Entrada | Consentimento do titular + credenciais Open Finance |
| Saida | Transacoes normalizadas no PostgreSQL |

### Modulo 2 — Categorizacao (Categorization)

| Atributo | Detalhe |
|----------|---------|
| Responsabilidade | Classificacao automatica de lancamentos, conciliacao bancaria, regras de categorizacao |
| Processamento | Sincrono (sob demanda) + batch pos-ingestao |
| Entrada | Transacoes normalizadas |
| Saida | Lancamentos categorizados e conciliados |

### Modulo 3 — IA e Projecoes (AI/Projections)

| Atributo | Detalhe |
|----------|---------|
| Responsabilidade | Projecoes de fluxo de caixa, deteccao de anomalias, alertas inteligentes |
| Processamento | Assincrono (chamadas a APIs de LLM) |
| Entrada | Historico de transacoes categorizadas |
| Saida | Projecoes, scores de anomalia, alertas |
| Dependencia externa | API de LLM (OpenAI / Claude) |

### Modulo 4 — Dashboard e Relatorios (Dashboard)

| Atributo | Detalhe |
|----------|---------|
| Responsabilidade | Visualizacao de dados, graficos, relatorios diarios/mensais, resumo executivo |
| Processamento | Sincrono (leitura de dados ja processados) |
| Entrada | Dados categorizados + projecoes |
| Saida | Interface React, PDFs de relatorios |

## Worker Assincrono — Ingestao

```
[Scheduler] ---> [BullMQ Queue] ---> [Ingestion Worker]
  (a cada 4h)       (Redis)          (processa jobs)
                                          |
                                          v
                                    [PostgreSQL]
                                    (transacoes normalizadas)
```

O worker de ingestao opera independentemente do servidor web. Jobs sao enfileirados pelo scheduler a cada 4 horas e processados em paralelo (uma fila por instituicao financeira conectada). Falhas sao retentadas com backoff exponencial.

## Padroes de Integracao

| Integracao | Padrao | Detalhe |
|-----------|--------|---------|
| Open Finance (Pluggy) | REST API + Webhook | Webhooks para notificacao de novos dados disponiveis |
| LLM (OpenAI/Claude) | REST API | Chamadas sob demanda com cache de respostas |
| Entre modulos internos | Chamada direta (in-process) | Sem overhead de rede; fronteira via interfaces TypeScript |
| Eventos internos | BullMQ | Ingestao concluida dispara categorizacao em cadeia |

---

## Plano de Escalabilidade

| Fase | Abordagem | Gatilho |
|------|-----------|---------|
| MVP (Fase 1) | Monolito modular, single instance + worker | Ate ~500 empresas |
| Fase 2 | Separacao do worker de ingestao como servico independente; replicas horizontais do monolito | Acima de 500 empresas ou latencia de ingestao > 30min |
| Fase 3 | Microservicos completos por modulo | Necessidade de escala independente por dominio |

A transicao de fases sera guiada por **metricas observadas** (latencia, throughput, custo de infra), nao por decisao prematura.
