Descritivo

O Data Hub é uma plataforma de dados que centraliza a extração, higienização e consolidação de informações oriundas de 15 sistemas transacionais distintos — entre ERPs legados, APIs de parceiros, planilhas e bancos relacionais espalhados por unidades de negócio. O objetivo é eliminar a fragmentação que hoje impede análises comparativas consistentes e atrasa decisões estratégicas.

A solução adota stack 100% Google Cloud Platform: o Sensedia API Gateway orquestra a extração dos dados nos sistemas-origem; os arquivos brutos aterrissam no Google Cloud Storage em formato raw (JSON/CSV/Parquet); o Cloud Pub/Sub atua como trigger para disparar o Apache Airflow (Cloud Composer), responsável pela limpeza, normalização e consolidação em uma tabela unificada na camada prata do BigQuery; em seguida o mesmo Airflow promove a consolidação para a camada ouro, onde 2 dashboards de consulta permitirão aos times de negócio explorar e filtrar os dados com autonomia.

Por que agora: a diretoria definiu consolidação de indicadores como OKR do semestre. Sem o Data Hub, o fechamento trimestral segue dependendo de planilhas manuais e ETLs ad-hoc mantidos por um único analista.
Problemas Atuais
5 dores mapeadas
Tempo de ciclo de dados: relatórios mensais consolidados demoram em média 8 dias úteis após o fechamento, com retrabalho recorrente por inconsistências entre fontes.
Fragmentação de fontes: os 15 sistemas transacionais não se comunicam; cada extração é manual, via export de CSV, consulta SQL direta ou API sem padronização. Não há catálogo unificado.
Bus factor crítico: a consolidação depende de 1 analista sênior que mantém scripts Python em laptop pessoal. Férias ou saída geram risco operacional alto.
Dashboards desatualizados: os paineis de BI existentes apontam para bases paralelas que divergem entre si — duas áreas olham o mesmo KPI e encontram valores diferentes.
Falta de trilha de auditoria: não há histórico versionado das transformações aplicadas aos dados; reprocessar um mês anterior exige reengenharia manual.
O Que Queremos Resolver

Estabelecer uma fonte única de verdade para os dados consolidados das 15 fontes, com atualização incremental automática, trilha versionada e acesso self-service para as áreas de negócio. Ao final do projeto, o fechamento consolidado deve sair de 8 dias úteis para até 24 horas, e o tempo de montagem de uma nova análise comparativa deve cair de 3 dias para menos de 30 minutos — sem depender de um único analista.

Tese de valor: governança de dados + velocidade analítica + redução de risco operacional. A escolha por GCP gerenciado (BigQuery, Cloud Composer, Pub/Sub) maximiza ROI ao evitar operar infraestrutura proprietária.

Descritivo do Cenário Recomendado
Entrega completa — 4 fases · Quick Win na Semana 10 · 24 semanas · 1.860h
Recomendado

Este cenário entrega o Data Hub end-to-end: pipeline automatizado das 15 fontes até a camada ouro no BigQuery, com 2 dashboards publicados em Power BI (consulta e filtros interativos). A arquitetura é 100% gerenciada em GCP, o que reduz carga operacional e concentra investimento em engenharia de dados.

Adota-se modelo medallion (raw → silver → gold) para separar responsabilidades: o GCS preserva o dado bruto (replay/auditoria), o Airflow aplica regras de limpeza e unificação na prata, e a camada ouro é modelada para consumo analítico. O Pub/Sub desacopla extração e processamento — cada arquivo novo no GCS dispara a DAG correspondente.

Por que é o cenário recomendado: cobre os 15 sistemas sem cortes, elimina o bus factor operacional, e deixa a base pronta para futuras extensões (ML, forecasting) sem reengenharia. A relação custo/risco/valor é a mais favorável das 3 opções avaliadas.

O Que Será Feito
  • Extração automatizada dos 15 sistemas via Sensedia
    1. Catálogo de conectores por tipo (SOAP, REST, JDBC, SFTP)
    2. Schedules e retentativas parametrizáveis no gateway
    3. Monitoramento centralizado de falhas de extração
  • Armazenamento raw no GCS
    1. Bucket por ambiente (dev/hml/prod) com lifecycle policy
    2. Particionamento por data de extração (yyyy/MM/dd)
    3. Criptografia em repouso (CMEK gerenciada)
  • Consolidação na camada prata (Airflow + BigQuery)
    1. DAGs disparadas por Pub/Sub (evento por arquivo novo)
    2. Regras de limpeza, deduplicação e normalização
    3. Tabela unificada com lineage e versionamento de schema
  • Modelagem da camada ouro
    1. Tabelas dimensionais (fato + dimensões conformadas)
    2. Views materializadas para consultas recorrentes
    3. Controle de acesso por área de negócio (IAM + row-level)
  • 2 dashboards em Power BI
    1. Dashboard Operacional — volumetria e qualidade por fonte
    2. Dashboard Analítico — KPIs consolidados com drill-down
    3. Treinamento e documentação de uso para 35+ usuários
O Que Não Será Feito
  • Modelos de Machine Learning
    1. Forecasting de séries temporais (forecast engine)
    2. Classificação/clusterização automatizada
    3. Anomalia detection em tempo real
  • Write-back para os sistemas de origem
    1. Não há gravação reversa nos 15 sistemas
    2. Correções aplicadas permanecem na camada prata/ouro
  • Plataforma de Self-Service Analytics
    1. Editor de queries para usuário final (ex: workspace próprio)
    2. Workspace individualizado de análise
  • Data Lake para dados não-estruturados
    1. Ingestão de imagens, PDFs, áudio
    2. OCR ou extração de documentos
  • Migração do BI legado atual
    1. Paineis Power BI existentes permanecem isolados
    2. Descontinuação acontece em projeto posterior
