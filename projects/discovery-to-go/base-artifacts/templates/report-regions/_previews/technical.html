Architecture Decision Records

REG-TECH-05
ADR-001 — Adotar PostgreSQL como banco principal Aceita 2025-01
Contexto
O FinTrack Pro precisa de um banco relacional com suporte a transacoes ACID para operacoes financeiras, capacidade de lidar com dados semi-estruturados (JSON) e ecossistema maduro de ferramentas.
Decisao
Utilizar PostgreSQL 16 como banco de dados principal, hospedado em AWS RDS com Multi-AZ.
Alternativas Consideradas
  • MySQL 8 — descartado por suporte inferior a JSON e funcionalidades analiticas
  • MongoDB — descartado por falta de garantias ACID nativas em transacoes multi-documento
  • CockroachDB — descartado por custo elevado e complexidade operacional para o volume atual
Consequencias
Garantias ACID robustas para operacoes financeiras
Suporte nativo a JSONB para dados flexiveis de configuracao
Amplo ecossistema de ferramentas e profissionais
Escalabilidade horizontal limitada — sharding manual se ultrapassar 1TB
ADR-002 — Autenticacao via Okta (SSO) Aceita 2025-01
Contexto
A organizacao ja possui licenca corporativa do Okta. Construir autenticacao custom demandaria 3+ sprints e nao atenderia requisitos de SSO empresarial.
Decisao
Uso da licenca corporativa existente do Okta com SAML 2.0 / OIDC para autenticacao e autorizacao.
Alternativas Consideradas
  • Build custom (JWT + bcrypt) — descartado por custo e risco de seguranca
  • Keycloak — descartado por complexidade operacional self-hosted
  • Auth0 — descartado por custo adicional com licenca Okta ja existente
Consequencias
Reduz 3 sprints de desenvolvimento
SSO corporativo pronto (SAML + OIDC)
Dependencia de vendor lock-in com Okta
ADR-003 — SQS como fila de mensagens Aceita 2025-02
Contexto
O sistema precisa de comunicacao assincrona entre Core Service e Notification Service. O volume atual (~10k mensagens/dia) nao justifica a complexidade do Kafka.
Decisao
Adotar AWS SQS como fila de mensagens serverless, integrada ao ecossistema AWS existente.
Alternativas Consideradas
  • Apache Kafka — descartado por over-engineering para o volume atual
  • RabbitMQ — descartado por custo operacional de self-hosting
  • AWS SNS + SQS — considerado para fan-out futuro, mas simplificado para SQS puro no MVP
Consequencias
Serverless — zero gestao de infraestrutura
Integracao nativa com AWS ECS e CloudWatch
Limitacao de 256KB por mensagem
Sem replay de eventos (diferente de Kafka)
