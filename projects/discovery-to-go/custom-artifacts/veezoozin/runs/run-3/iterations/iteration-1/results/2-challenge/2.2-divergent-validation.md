## 2. Premissas Desafiadas

### P1 — "Bus factor = 1 é gerenciável com documentação e auto-healing"

**O que o discovery assume:** Bloco 1.4 propõe 3 camadas de mitigação: auto-healing (Cloud Run restart), documentação excepcional (ADRs, runbooks), e freelancer em standby. Custo: R$ 0-1K/mês.

**O desafio:**

O bus factor = 1 não é apenas um risco técnico — é um **risco existencial para o negócio**:

1. **Doença prolongada.** Se Fabio ficar incapacitado por 4+ semanas (doença séria, acidente, burnout severo), os tenants Pro que pagam R$ 497/mês ficam sem suporte. O SLA de 99.5% é violado. Clientes cancelam.

2. **Freelancer standby é teoria.** Encontrar um dev que conheça a stack (Python + FastAPI + GCP + NL-to-SQL + multi-tenant) e esteja disponível em <48h para assumir um codebase que nunca viu é **irrealista**. Mesmo com ADRs, o ramp-up é de 2-4 semanas.

3. **O auto-healing resolve infra, não produto.** Cloud Run reinicia containers. Mas se houver um bug no pipeline NL-to-SQL que gera SQL incorreto para um schema específico, o auto-healing não resolve — precisa de debugging humano.

4. **Burnout é probabilidade, não risco.** Com 1 pessoa fazendo dev + ops + suporte + PM + DPO + vendas (beta program), burnout em 6-12 meses é **provável**, não apenas possível. Os rituais anti-burnout (no-deploy Friday, férias trimestrais) são boas práticas, mas insuficientes se a carga base for insustentável.

**Impacto se a premissa falhar:**
- 4 semanas de indisponibilidade = ~50% dos tenants Pro cancelam (sem suporte, sem fixes)
- MRR cai pela metade, recuperação leva 6+ meses
- Reputação danificada — "o produto que parou de funcionar"

**Recomendação:** Além do plano atual, implementar:
- Conta de poupança "seguro do produto" com 3 meses de custo fixo (R$ 51K) antes do launch
- Acordo formal com 1-2 devs freelancer (contrato, NDA, acesso read-only ao repo) — não apenas "identificar"
- Automação agressiva de operações: 100% das tarefas de suporte Free devem ser chatbot/self-service antes do launch
- Health check pessoal mensal: se score de burnout >7/10 por 2 meses consecutivos, reduzir escopo (pausar Free, focar em Pro)

### P2 — "BYOK é transparente para o usuário final"

**O que o discovery assume:** Bloco 1.2 diz que Marina (Gestora) "nem sabe que existe LLM por trás". A API key é gerenciada pela Carla (Admin TI) e o custo é monitorado pelo Lucas (Admin Tenant).

**O desafio:**

BYOK cria uma cadeia de dependência **fora do controle do Veezoozin**:

1. **Key expirada = produto para.** Se a API key do tenant expira (créditos esgotados, cartão de crédito rejeitado pelo provider), todas as queries falham. Marina vê "erro" e não entende por quê. O healthcheck diário detecta, mas o delay pode ser de 12-24h.

2. **Custo surpresa assusta o CFO.** Se Rafael (Analista) faz 200 queries/dia com Claude (modelo mais caro), o custo de LLM pode ser R$ 500-2.000/mês — **mais que o plano Pro do Veezoozin**. O dashboard de consumo ajuda, mas o "susto da fatura" da Anthropic/Google pode gerar percepção de que "o Veezoozin é caro" mesmo que o custo seja da API.

3. **Suporte técnico de LLM recai sobre o Veezoozin.** Quando a API do Claude retorna erro 429 (rate limit) ou 500 (server error), quem o tenant liga? Não para a Anthropic — liga para o Veezoozin. Fabio vira "suporte de terceiro" para 3 provedores de LLM que ele não controla.

4. **Qualidade varia por provider.** Se Tenant A usa Claude (excelente em SQL) e Tenant B usa Gemini Flash (mais barato, menor precisão), a experiência do produto é inconsistente. O Veezoozin leva a culpa pela diferença de qualidade do LLM.

**Recomendação:**
- Implementar "recommended models" por caso de uso: "Para SQL generation, recomendamos Claude Sonnet ou Gemini Pro"
- Estimativa de custo mensal de LLM no momento do signup ("com seu uso estimado, a API key custará ~R$ X/mês")
- Circuit breaker com mensagem amigável: "Sua API key precisa de atenção. Contacte seu administrador."
- Dashboard de comparação de qualidade por provider (anonimizado) para guidar escolha

### P3 — "Precisão NL-to-SQL > 85% é atingível no MVP"

**O que o discovery assume:** KR1.1 define >85% de queries corretas na primeira tentativa. O pipeline RAG com glossário por tenant, few-shot examples, e schema mapping seria suficiente.

**O desafio:**

A meta de 85% é **ambiciosa para schemas desconhecidos**:

1. **Spider benchmark (2024-2025):** Melhores modelos atingem ~80% em exact match em schemas **conhecidos** com treinamento extensivo. O Veezoozin usa schemas **desconhecidos** (cada tenant traz o seu) com **zero fine-tuning** (RAG only).

2. **Schema quality varia dramaticamente.** Se o tenant tem tabelas bem nomeadas (`orders`, `customers`, `revenue_by_region`), a precisão pode ser 85%+. Se tem `tb_01`, `col_a`, `col_b`, a precisão cai para 50-60%, mesmo com glossário.

3. **Multi-idioma adiciona ambiguidade.** Pergunta em PT-BR sobre schema em EN-US: "Qual o faturamento por região?" → precisa mapear "faturamento" para `revenue` e "região" para `region`. Se o glossário não tiver esse mapeamento, o LLM adivinha.

4. **Queries complexas são muito mais difíceis.** "Faturamento por região no último trimestre" (1 tabela, 1 filtro) é fácil. "Qual produto teve maior crescimento de vendas comparando Q1 vs Q2 por região, excluindo devoluções?" (JOINs, subqueries, filtros compostos) é dramaticamente mais difícil.

**O kill criteria correto é definido (D4.8):** Precisão <70% após tuning → pivotar para query builder assistido. Isso é bom. Mas a meta de 85% no MVP (mês 4) provavelmente deveria ser 75-80%, com 85% como meta do mês 8.

**Recomendação:**
- Definir meta escalonada: >75% (mês 4), >80% (mês 6), >85% (mês 8)
- Implementar "confidence score" — se o LLM tem <70% de confiança no SQL gerado, pedir confirmação do usuário antes de executar
- "Did you mean?" para queries ambíguas — oferecer 2-3 interpretações
- Investir no feedback loop (thumbs up/down + correção manual) como acelerador de precisão

## 3. Pontos Cegos (Blind Spots)

### B1 — Sustentabilidade de Longo Prazo com 1 Pessoa

O discovery projeta milestones de contratação (SRE em R$ 25K MRR, CS em R$ 40K MRR). Mas entre o lançamento (mês 4) e a 1a contratação (mês ~14-18), são **10-14 meses de operação solo**. Durante esse período:

- Fabio é dev, DevOps, SRE, PM, suporte, DPO, vendas (beta program), e marketing (PLG)
- Cada tenant Enterprise adiciona ~4h/mês de suporte dedicado
- Cada incidente SEV-1 consome 2-4h de investigação + fix + post-mortem
- Cada feature request entra na fila sem priorização de PM

**Pergunta não respondida:** Qual é a capacidade real de Fabio? Se são 160h/mês de trabalho produtivo, e suporte + ops + marketing consomem 40h/mês, restam 120h para desenvolvimento. Com Claude Code (3-5x aceleração), equivale a ~400h de output. É suficiente? Ninguém calculou.

### B2 — Exit Strategy

O discovery não menciona exit strategy. Cenários:

1. **Bootstrapping eterno:** Fabio opera o Veezoozin indefinidamente. Viável financeiramente (margem 86%), mas sem crescimento explosivo (sem time, sem investimento)
2. **Aquisição:** Metabase, Looker, ou similar compra o Veezoozin pela tecnologia NL-to-SQL + base de clientes
3. **Investimento:** Pre-seed/Seed para crescer time e acelerar

Sem exit strategy definida, as decisões arquiteturais podem otimizar para o cenário errado.

### B3 — Concorrência de BigQuery Nativo

O Google pode lançar uma feature de "Natural Language queries" nativo no BigQuery a qualquer momento. Se isso acontecer:
- É gratuito para clientes BigQuery (o público-alvo exato do Veezoozin)
- Tem contexto nativo do schema (sem necessidade de mapeamento)
- Tem integração com Looker Studio (visualização)

O Veezoozin precisa de um plano de contingência: se BigQuery ganhar NL nativo, qual é o diferencial restante? A resposta é: glossário de negócio + histórico + multi-idioma + insights contextualizados. Mas precisa ser articulado.

### B4 — Validação com Clientes Reais

O discovery inteiro é baseado em simulação ([BRIEFING], [INFERENCE], [RECOMENDAÇÃO]). Nenhum cliente real validou:
- Disposição a pagar R$ 497/mês
- Aceitabilidade do modelo BYOK
- Onboarding de 45 minutos como realista
- Precisão de NL-to-SQL como suficiente

O beta program (D4.6) é a resposta correta, mas acontece **após** o MVP ser construído. Idealmente, 3-5 LOIs (letters of intent) deveriam existir antes de iniciar desenvolvimento.

### B5 — Custo Oculto de BYOK Multi-Provider

Suportar 3 provedores de LLM (Claude, Gemini, OpenAI) significa:
- 3 SDKs para manter atualizados
- 3 formatos de prompt (cada modelo tem quirks)
- 3 modelos de pricing (tokens vs characters)
- 3 modelos de rate limiting
- Testes de regressão quando qualquer provider muda API

O adapter pattern (D5.6) é a solução correta, mas o custo de manutenção de 3 integrações não está contabilizado no roadmap de 12 semanas.

## 4. Stress-test Financeiro

### O cenário base sob estresse moderado

O run-3 tem um cenário base **viável**. Mas vamos stress-testar com premissas moderadamente pessimistas (não extremas):

| Premissa do cenário base | Ajuste pessimista moderado | Impacto |
|--------------------------|:-------------------------:|---------|
| 27 Pro no mês 16 | 18 Pro no mês 16 (33% menos) | MRR R$ 8.946 + Enterprise → break-even atrasa 4-6 meses |
| Churn 5%/mês (implícito) | Churn 8%/mês (turnover de PME) | MRR estabiliza mais cedo, break-even atrasa 3 meses |
| Custo variável R$ 70/tenant | R$ 90/tenant (sem cache efetivo) | Custo total sobe R$ 600/mês com 30 tenants |
| 1a contratação mês 14 | 1a contratação mês 12 (burnout) | Custo fixo sobe 2 meses antes |
| 3 Enterprise no mês 19 | 1 Enterprise no mês 19 | MRR -R$ 5K/mês |

**Cenário pessimista moderado combinado (3 de 5 premissas falham):**

| Métrica | Cenário base | Pessimista moderado |
|---------|:----------:|:------------------:|
| Break-even operacional | Mês 19 | Mês 26-28 |
| Investimento até break-even | R$ 184K | ~R$ 280K |
| MRR no mês 24 | R$ 49.836 | ~R$ 30.000 |
| ROI 3 anos | +R$ 155K | ~+R$ 20K (marginal) |

**Veredicto do stress test:** Mesmo no cenário pessimista moderado, o projeto **permanece viável** (break-even atingido, ROI positivo em 3 anos). A diferença entre run-2 e run-3 é que no run-2 o cenário pessimista matava o projeto; no run-3, apenas atrasa o break-even. Isso é resiliência financeira real.

## 5. Desafio ao Consenso

### "1 dev + Claude Code equivale a ~3 devs"

**Contra-argumento:** Claude Code acelera **coding** 3-5x, mas não acelera:
- Reuniões com clientes beta (onboarding, feedback)
- Debugging de issues de produção (investigação, contexto)
- Decisões de produto (priorização, roadmap)
- Suporte ao cliente (comunicação humana)
- Marketing e vendas (posts LinkedIn, demos)
- Compliance (DPA, LGPD, resposta a titulares)

O gargalo não é velocidade de código — é **bandwidth de atenção humana**. Claude Code resolve o gargalo de código, mas cria ilusão de capacidade em áreas que exigem presença humana.

### "O BYOK é uma vantagem competitiva"

**Contra-argumento parcial:** BYOK é financeiramente brilhante (margem 86-94%). Mas para o usuário final, é **fricção**:
- "Preciso de uma API key de IA para usar uma ferramenta de dados?" — confuso para Marina
- "Por que estou pagando R$ 497/mês E mais R$ 200/mês de API?" — percepção de custo duplicado
- Concorrentes que absorvem custo de LLM parecem "mais simples" mesmo sendo mais caros no total

BYOK é vantagem para o **negócio**, não necessariamente para o **usuário**. A comunicação precisa ser clara: "Você controla seu provedor de IA e seus custos — sem mark-up, sem vendor lock-in."

### "O mercado de NL-to-SQL é grande o suficiente"

**Contra-argumento:** O MVP suporta apenas BigQuery. "Empresas brasileiras com dados em BigQuery" é um nicho dentro de um nicho. Sem TAM/SAM, não sabemos se existem 300 empresas assim (necessário para cenário de escala) ou 30 (suficiente apenas para break-even).

## 8. Veredicto

### APROVADO COM RESSALVAS

O discovery do Veezoozin run-3 é um **salto qualitativo** em relação ao run-2. O modelo financeiro é viável, a consistência é alta, e os especialistas são proativos. O 10th Man reconhece que os problemas graves do run-2 (viabilidade negativa, inconsistências de pricing/breakeven, mitigações genéricas) foram resolvidos.

**Ressalvas (não bloqueantes, mas devem ser monitoradas):**

1. **Bus factor = 1 permanece como risco existencial.** As mitigações propostas (docs + auto-healing + freelancer) são necessárias mas insuficientes para uma ausência prolongada. Recomenda-se: fundo de emergência de 3 meses + acordo formal com freelancer.

2. **BYOK cria fricção que precisa ser gerenciada.** O modelo é financeiramente excelente, mas a comunicação ao cliente precisa ser clara sobre custos e benefícios. Estimativa de custo de LLM no signup e "recommended models" são essenciais.

3. **Precisão NL-to-SQL deve ter meta escalonada.** >75% (MVP), >80% (mês 6), >85% (mês 8). A meta de 85% no mês 4 é otimista para schemas desconhecidos.

4. **Sustentabilidade operacional solo não foi stress-testada.** O gargalo real não é código (Claude Code resolve) — é bandwidth de atenção humana para suporte, vendas, compliance e decisões de produto.

5. **TAM/SAM/SOM ausente.** O volume necessário é baixo (~30 clientes), mas sem dimensionamento de mercado, não há como validar se o cenário de escala é realista.

**Condições atendidas vs run-2:**
- [x] Viabilidade financeira demonstrada (break-even mês 19, margem 86-94%)
- [x] Consistência inter-blocos (pricing e breakeven alinhados)
- [x] Mitigações detalhadas (P16 atendido)
- [x] Kill criteria definidos
- [ ] TAM/SAM/SOM (pendente)
- [ ] Validação com clientes reais (beta program planejado para mês 10+)
- [ ] Exit strategy (não endereçado)
