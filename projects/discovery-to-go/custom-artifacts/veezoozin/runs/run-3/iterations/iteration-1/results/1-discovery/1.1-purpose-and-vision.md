# Bloco #1 — Propósito e Visão

> Resultado da entrevista simulada (Fase 1 — Discovery) entre PO e Customer simulado, cobrindo propósito do produto, problema central, visão de futuro, posicionamento competitivo e modelo de negócio. **Run-3 incorpora mudanças estruturais:** equipe de 1 dev + Claude Code, BYOK (custo LLM do tenant), infraestrutura 100% pay-per-use.

## Parte 1 — Dados Coletados

### 1.1. Problema Central

O Veezoozin resolve um problema estrutural presente em empresas de todos os portes: **dados valiosos existem, mas estão presos atrás de barreiras técnicas**.

O problema se manifesta em quatro dimensões:

| Dimensão | Descrição | Impacto | Fonte |
|----------|-----------|---------|-------|
| **Barreira técnica** | Gestores e executivos dependem de times de dados para obter respostas. Uma pergunta simples vira um ticket que leva dias. | Decisões atrasadas, custo operacional alto | [BRIEFING] |
| **Falta de contexto de negócio** | Ferramentas de BI genéricas não entendem o vocabulário do cliente. "Churn" significa coisas diferentes em cada empresa. | Respostas imprecisas, retrabalho | [BRIEFING] |
| **Dados sem ação** | O usuário recebe tabelas brutas sem insights. Falta a camada de interpretação: gráficos, tendências, anomalias, recomendações. | Dados acessados mas não utilizados | [BRIEFING] |
| **Multi-idioma** | Empresas latinas operam em PT-BR, EN-US e Espanhol. O sistema precisa entender perguntas em qualquer idioma. | Exclusão de usuários, adoção limitada | [BRIEFING] |

**Síntese:** O problema não é falta de dados — é falta de acesso democratizado, contextualizado e acionável a esses dados.

### 1.2. Job to be Done (JTBD)

> "Quando eu tenho uma dúvida sobre o meu negócio, quero fazer uma pergunta em linguagem natural e receber uma resposta visual, contextualizada e acionável em segundos — sem depender de ninguém técnico."

| Persona | Job to be Done | Fonte |
|---------|----------------|-------|
| Gestor / Executivo | Obter respostas rápidas sobre KPIs de negócio para tomar decisões informadas no dia a dia | [BRIEFING] |
| Analista de negócio | Cruzar dados de múltiplas fontes, comparar períodos e identificar tendências sem escrever SQL | [BRIEFING] |
| Admin do tenant | Ensinar o sistema sobre o domínio de negócio da empresa para que as respostas sejam precisas | [BRIEFING] |
| Admin de TI | Conectar bancos de dados com segurança e controlar quem acessa o quê | [BRIEFING] |

### 1.3. Visão do Produto

**Declaração de visão:**

> Ser a camada de inteligência entre as pessoas e seus dados — tornando qualquer colaborador capaz de consultar, entender e agir sobre os dados da empresa, em qualquer idioma, com contexto de negócio.

**Posicionamento:** "Pergunte para seus dados" — democratização do acesso a dados via linguagem natural, com contexto de negócio por tenant, multi-idioma nativo (PT-BR/EN/ES), e integração com fontes de conhecimento externas via MCP.

### 1.4. Visão em Camadas

```
Camada 1 (MVP — 4 meses)  → Pergunta NL → SQL → Visualização (BigQuery, 3 idiomas)
Camada 2 (Fase 2 — +3m)   → Multi-banco (PostgreSQL, MySQL) + MCPs + RAG externo
Camada 3 (Fase 3 — +3m)   → Análises preditivas, alertas automáticos, agentes autônomos
```

## Parte 2 — Análise do Especialista

### Avaliação da proposta de valor

O Veezoozin ataca um problema real e bem dimensionado. A combinação de NL-to-SQL + contexto por tenant + multi-idioma cria uma proposição de valor diferenciada. No entanto, a mudança para **1 dev + Claude Code** e **BYOK** altera fundamentalmente a equação econômica — e isso é uma **vantagem**, não uma limitação.

**Pontos fortes do novo modelo:**

1. **Break-even radicalmente baixo.** Com custo fixo de R$ 17K/mês (contra R$ 75K+ de um time convencional), o produto precisa de apenas ~25 clientes Pro para se pagar. Isso significa que a validação de mercado pode acontecer **antes** de queimar caixa.

2. **BYOK elimina o maior risco financeiro de SaaS com IA.** Em produtos que absorvem custo de LLM, cada query é um custo variável. Com BYOK, o custo variável por tenant cai para ~R$ 40-100/mês (apenas infra GCP). A margem bruta por tenant fica em 80-90%.

3. **Pay-per-use alinha custo com receita.** Cloud Run, BigQuery e Firestore cobram por uso. Sem clientes = custo quase zero. Isso elimina o "death valley" típico de SaaS onde a infra custa antes de gerar receita.

4. **Velocidade de decisão.** 1 pessoa com autonomia total executa mais rápido que 5 pessoas com governance. Claude Code multiplica a capacidade de output 3-5x.

**Pontos de atenção:**

1. **Bus factor = 1 é o risco existencial.** Se Fabio ficar indisponível, o produto para. Mitigação: documentação excepcional, código limpo, arquitetura simples.

2. **BYOK cria fricção no onboarding.** O usuário do plano Free precisa ter uma API key de LLM. Isso exclui 90% dos leads que "só querem testar". Precisa de uma solução para o tier Free.

3. **Precisão de NL-to-SQL é o make-or-break.** Se a primeira query do usuário der errado, ele não volta. O glossário por tenant é a solução, mas depende de configuração manual.

### Avaliação da diferenciação competitiva

O triângulo competitivo (contexto por tenant + multi-idioma + MCP) é sólido. No entanto, o mercado de NL-to-SQL está aquecido — Databricks, Snowflake e até o Google BigQuery estão adicionando capacidades de linguagem natural. A janela de oportunidade para o Veezoozin é de **12-18 meses**.

O **moat real** não é a tecnologia (NL-to-SQL commoditiza) — é o **contexto acumulado por tenant**. Quanto mais o tenant usa, mais o sistema aprende sobre o domínio, e mais difícil é trocar. Isso é o network effect de dados do cliente.

## Parte 3 — Recomendações

### R1. Resolver o problema de BYOK no tier Free

**Problema:** Se o plano Free exige que o usuário traga sua própria API key, a barreira de entrada é muito alta para a maioria dos leads.

**Recomendação:** Oferecer um **pool de créditos LLM subsidiados** no plano Free (ex: 50 queries/mês com API key compartilhada da mAInd Tech). Custo estimado: ~R$ 0,50-1,00/usuário Free/mês (assumindo queries simples). Se 100 usuários Free custam R$ 100/mês, é um custo de aquisição aceitável. A partir do Pro, é obrigatório BYOK.

**Alternativa:** Integração com LLMs gratuitos ou de custo ultra-baixo (Gemini Flash, Llama via Groq) para o tier Free, com performance reduzida mas funcional.

### R2. Modelo de precificação sugerido

| Tier | Preço/mês | LLM | Limite de queries | Fontes de dados | Observação |
|------|----------:|-----|------------------:|----------------:|------------|
| **Free** | R$ 0 | Pool compartilhado (50 queries/mês) | 50/mês | 1 BigQuery dataset | Sem glossário custom, sem MCP, sem export |
| **Pro** | R$ 497 | BYOK obrigatório | Ilimitado | Até 10 datasets | Glossário, export PDF/HTML, histórico completo |
| **Enterprise** | R$ 2.497+ | BYOK obrigatório | Ilimitado | Ilimitado | SSO, SLA 99.9%, suporte dedicado, RBAC avançado |

**Justificativa do preço Pro em R$ 497:**
- Break-even com ~25 Pro + 2 Enterprise = ~R$ 17K/mês
- 25 x R$ 497 = R$ 12.425 + 2 x R$ 2.497 = R$ 4.994 = R$ 17.419 (cobre custo fixo + variável)
- Preço competitivo para PME: ThoughtSpot custa US$ 1.250+/mês

### R3. Priorizar feedback loop sobre precisão

A precisão de NL-to-SQL é o KR mais crítico. Recomendo um **ciclo de feedback em 3 camadas**:

1. **Thumbs up/down** em cada resposta — coleta sinal binário
2. **"A query estava certa?"** — mostra o SQL gerado, permite correção manual
3. **Aprendizado por tenant** — queries corrigidas alimentam o contexto, melhorando a próxima execução

Isso transforma o uso em treinamento contínuo e cria o moat de dados.

### R4. Definir "aha! moment" como métrica norte do MVP

O "aha! moment" é: **primeira query que retorna um gráfico com insight correto em menos de 5 segundos**. Toda a experiência de onboarding deve ser otimizada para esse momento.

Métrica: **% de novos tenants que atingem o "aha! moment" em menos de 45 minutos após signup**.

### R5. Documentação como seguro contra bus factor

Com bus factor = 1, a documentação é literalmente seguro de vida do produto. Recomendo:

- ADRs (Architecture Decision Records) para toda decisão técnica
- README operacional: como fazer deploy, rollback, debug
- Runbooks para incidentes comuns
- Claude Code como "segundo cérebro" — o histórico de sessões é documentação viva
