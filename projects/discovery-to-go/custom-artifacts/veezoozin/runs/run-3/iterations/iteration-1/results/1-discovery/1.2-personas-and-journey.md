## Parte 2 — Análise do Especialista

### Validação das Personas

As 4 personas cobrem o espectro completo de um SaaS B2B:

1. **Marina (Gestora)** — Buyer persona. Sente a dor, tem budget, decide.
2. **Rafael (Analista)** — User persona. Usa intensivamente, valida qualidade técnica.
3. **Lucas (Admin Tenant)** — Enabler persona. Sem ele, o produto não funciona.
4. **Carla (Admin TI)** — Gatekeeper persona. Pode bloquear a adoção.

**Gap identificado:** Falta uma persona **"Champion interno"** — alguém que evangeliza o Veezoozin dentro da empresa cliente. Em SaaS B2B, o champion é quem garante renovação e expansão. Provavelmente é o Rafael (power user satisfeito) ou o Lucas (que vê a redução de tickets).

### Impacto do BYOK nas Personas

O modelo BYOK cria dinâmicas específicas por persona:

| Persona | Impacto do BYOK | Risco |
|---------|-----------------|-------|
| Marina | Transparente — nem sabe que existe | Nenhum |
| Rafael | Pode se preocupar com "custo por query" se souber | Baixo — mitigar com dashboard de consumo |
| Lucas | Responsável por monitorar consumo | Médio — precisa de alertas de gastos |
| Carla | Gerencia API key como secret | Alto — precisa de rotação, vault, auditoria |

### Análise da Jornada de Onboarding

O time-to-value de ~45 minutos é **ambicioso mas viável**, com premissas:

1. O mapeamento automático de schema precisa funcionar bem (80%+ de acerto)
2. O glossário mínimo (5-10 termos) já melhora significativamente a precisão
3. A primeira query precisa ser "fácil" — perguntas sobre faturamento, contagem, agregações simples

**Risco crítico:** Se a primeira query falhar, o "aha! moment" vira "abandonment moment". A taxa de tolerância a erro na primeira experiência é zero.

### Consistência com Bloco #1

| Decisão do Bloco #1 | Como impacta personas | Status |
|---------------------|----------------------|--------|
| Modelo Free/Pro/Enterprise | Onboarding diferenciado por tier | Consistente |
| BYOK a partir do Pro | Free com pool compartilhado resolve barreira | Consistente |
| Break-even ~25 Pro | Preço R$ 497 viável para PMEs | Consistente |
| 1 dev + Claude Code | Onboarding Enterprise white-glove limita escala | Ponto de atenção |

## Parte 3 — Recomendações

### R1. Implementar "Guided First Query" no onboarding

**Problema:** Se a primeira query falhar, o usuário abandona.

**Recomendação:** Após o mapeamento de schema, o sistema deve **sugerir a primeira query** baseado nas tabelas encontradas. Ex: "Encontrei uma tabela de vendas com coluna de região e data. Quer ver o faturamento por região no último trimestre?" O usuário só precisa clicar "Sim".

**Impacto:** Transforma o "aha! moment" de dependente da criatividade do usuário para **garantido pelo sistema**.

### R2. Dashboard de consumo de LLM por tenant

**Problema:** O modelo BYOK transfere custo ao tenant, mas sem visibilidade o tenant pode se assustar com a fatura da API.

**Recomendação:** Painel no admin do tenant mostrando:
- Queries realizadas no mês
- Estimativa de custo de LLM (em R$ ou tokens)
- Tendência de consumo
- Alertas configuráveis (ex: "80% do budget mensal atingido")

**Persona beneficiada:** Lucas (Admin Tenant) e o financeiro invisível.

### R3. Criar persona "Champion" como métrica de produto

**Recomendação:** Monitorar qual usuário mais usa o Veezoozin dentro de cada tenant e transformá-lo em "Champion" oficial. Benefícios:
- Acesso beta a features novas
- Canal direto de feedback
- Badge no perfil

**Por que importa:** O Champion é quem garante renovação. Em SaaS B2B, 80% das renovações dependem de ter um champion ativo.

### R4. Onboarding Enterprise requer processo escalável

**Problema:** Com 1 dev + Claude Code, white-glove onboarding para Enterprise não escala. Se houver 3 clientes Enterprise simultâneos, Fabio gasta 100% do tempo em onboarding.

**Recomendação:** Criar um **playbook de onboarding automatizado** que cubra 80% do processo:
1. Wizard self-service com validação automática de conexão
2. Mapeamento de schema com sugestões de glossário (IA)
3. Chamada de 30 min (não 2h) apenas para revisão e ajustes finos
4. Documentação video-first (Loom) para processos repetíveis

**Milestone para white-glove completo:** Só oferecer consultoria dedicada de 2h quando receita Enterprise justificar contratação de customer success (>R$ 30K MRR).

### R5. Controle de acesso por campo/registro desde o MVP

**Problema:** O briefing menciona "possibilidades de controlar acesso em nível de registro/campo". A persona Carla (Gatekeeper) exige isso para aprovar o produto em empresas maiores.

**Recomendação:** No MVP, implementar RBAC básico:
- **Nível 1 (MVP):** Roles por dataset — Admin, Viewer, Analyst
- **Nível 2 (Fase 2):** Filtros por registro (row-level security no BigQuery)
- **Nível 3 (Enterprise):** Column-level masking (PII, salários, dados sensíveis)

Carla não aprova sem pelo menos Nível 1.
