### 1.2. Pipeline NL-to-SQL — Arquitetura de Processamento

> **Referência Blueprint AI/ML — Componente 2:** Tipo de problema: NL-to-SQL (geração de código). Abordagem: LLM via API com RAG contextual.

O pipeline NL-to-SQL é o **core técnico** do Veezoozin. Converte uma pergunta em linguagem natural em SQL válido para BigQuery, executa em sandbox read-only, e retorna resultado visual com insights.

```
┌────────────────────────────────────────────────────────────────────┐
│                    PIPELINE NL-to-SQL                               │
│                                                                     │
│  ┌──────────┐   ┌──────────────┐   ┌──────────────┐   ┌─────────┐ │
│  │ 1. Input  │──▶│ 2. Contexto  │──▶│ 3. Geração   │──▶│ 4. SQL  │ │
│  │ NL Query  │   │ Enriquecido  │   │    de SQL     │   │ Sandbox │ │
│  │ (PT/EN/ES)│   │ (RAG + Gloss)│   │  (LLM BYOK)  │   │ Execute │ │
│  └──────────┘   └──────────────┘   └──────────────┘   └─────────┘ │
│       │                │                    │                │      │
│       ▼                ▼                    ▼                ▼      │
│  Detecção de      Embedding da         Prompt com:       Validação:│
│  idioma (auto)    query → busca        - Schema           - Read   │
│                   no vector store      - Glossário          only   │
│                   do tenant            - Exemplos         - Timeout│
│                                        - Idioma           - Rows  │
│                                                             limit  │
│       ┌──────────────┐   ┌──────────────┐                         │
│       │ 5. Output    │◀──│ 6. Insight   │                         │
│       │    Visual    │   │   Generation │                         │
│       │ (Chart.js)   │   │  (LLM BYOK) │                         │
│       └──────────────┘   └──────────────┘                         │
└────────────────────────────────────────────────────────────────────┘
```

#### Detalhamento de cada etapa

| Etapa | Input | Processo | Output | Latência alvo | Fonte |
|:-----:|-------|---------|--------|:-------------:|-------|
| 1 | Pergunta NL do usuário | Detecção de idioma, normalização, tokenização | Query normalizada + idioma detectado | <100ms | [INFERENCE] |
| 2 | Query normalizada | Embedding da query → busca vetorial no contexto do tenant (schema + glossário + histórico) | Prompt enriquecido com contexto relevante | <500ms | [INFERENCE] |
| 3 | Prompt enriquecido | Chamada LLM (BYOK) com prompt estruturado: schema, glossário, exemplos, regras SQL | SQL BigQuery válido | <2s | [BRIEFING] |
| 4 | SQL gerado | Validação (read-only, sem DDL/DML), sandbox com timeout (30s) e limit (10K rows) | Resultset (dados tabulares) | <2s | [BRIEFING] |
| 5 | Resultset | Seleção automática de tipo de gráfico (bar, line, pie, scatter) baseado nos dados | Gráfico interativo | <300ms | [INFERENCE] |
| 6 | Resultset + query original | Chamada LLM (BYOK) para gerar insight textual sobre os dados | Insight + sugestão de próxima pergunta | <1s | [INFERENCE] |
| **Total** | — | — | — | **<5s** | [BRIEFING] |

### 1.3. Multi-Tenant Strategy

> **Referência Blueprint SaaS — Componente 2:** Multi-tenant strategy, isolamento, escalabilidade.

| Aspecto | Decisão | Justificativa | Fonte |
|---------|---------|---------------|-------|
| **Modelo de tenancy** | Row-level isolation (Cloud SQL) + dataset isolation (BigQuery) | Row-level no metadata (custo mínimo, simples para 1 dev). BigQuery é naturalmente isolado por dataset/project do tenant. | [INFERENCE] |
| **Identificação do tenant** | `tenant_id` em todas as tabelas de metadata + middleware obrigatório | Toda query de metadata inclui `WHERE tenant_id = ?`. Sem exceção. | [INFERENCE] |

| **Credenciais de banco** | Service account do tenant, armazenada criptografada no Secret Manager | Cada tenant fornece sua service account com read-only no BigQuery | [INFERENCE] |
| **API keys LLM (BYOK)** | Criptografadas no Secret Manager, por tenant | Chave do tenant nunca armazenada em plaintext. Acessível apenas em runtime. | [BRIEFING] |
| **Isolamento de execução** | Sandbox por query: timeout 30s, limit 10K rows, read-only | Previne queries destrutivas, abusivas ou infinitas | [BRIEFING] |
| **Cross-tenant leakage** | Impossível por design — cada tenant acessa apenas seu BigQuery | Tenant A não tem service account do Tenant B | [INFERENCE] |

## Parte 2 — Análise do Especialista

### Avaliação da Stack

A stack escolhida é **coerente com as restrições do projeto**:

1. **GCP-native é a decisão correta.** Com créditos e incentivos, cada serviço GCP é essencialmente gratuito no início. Cloud Run + BigQuery + Firestore formam a "holy trinity" do serverless GCP — pay-per-use real, sem custos ociosos.

2. **Python + FastAPI é a escolha certa para NL-to-SQL.** O ecossistema de IA em Python é incomparável. LangChain, SDKs de LLM, Vertex AI — tudo é Python-first. FastAPI adiciona tipagem e performance sem sacrificar produtividade.

3. **Next.js + Shadcn é pragmático para 1 dev.** Sem designer, usar uma component library pronta (Shadcn/Radix) reduz 80% do trabalho de UI. Next.js com SSR é bom para SEO da landing page.

4. **Firebase Auth é a escolha correta para o MVP.** Implementar auth from scratch para 1 dev é suicídio. Firebase Auth resolve OAuth2, email/password, 2FA e SSO com configuração mínima.

### Avaliação do Pipeline NL-to-SQL

O pipeline proposto é **sólido mas com ponto de risco crítico**: a qualidade do prompt para o LLM.

**Ponto forte:** A abordagem RAG (schema + glossário + exemplos) é superior a fine-tuning para este caso. Razões:
- Não precisa treinar modelo próprio (equipe de 1)
- Contexto por tenant é dinâmico (muda com glossário)
- Funciona com qualquer LLM (BYOK multi-provider)
- Custo zero de re-treinamento

**Ponto de risco:** A etapa 3 (geração de SQL) é black-box. Se o LLM gerar SQL errado, o sistema precisa de múltiplas camadas de defesa:
1. **Dry-run** no BigQuery (valida SQL sem executar)
2. **Parser** que rejeita DDL/DML
3. **Feedback do usuário** ("A query estava certa?") alimenta exemplos futuros
4. **Retry com contexto** — se o SQL falhar, re-gerar com mensagem de erro como contexto

### Avaliação do BYOK

O modelo BYOK é **tecnicamente viável e financeiramente brilhante**, mas cria complexidade operacional:

1. **Trânsito de chaves:** A API key do tenant transita pelo sistema do Veezoozin. Mesmo que fique apenas em memória, isso é um vetor de ataque. Mitigação: Secret Manager + audit log + nunca logar a key.

2. **Multi-provider:** Suportar Claude, Gemini e OpenAI significa manter 3 SDKs, 3 formatos de prompt, 3 modelos de erro. Recomendo um **adapter pattern** — interface unificada com implementações por provider.

3. **Degradação graceful:** Se a key do tenant falhar mid-session, o sistema precisa retornar erro amigável sem expor detalhes da API. Nunca mostrar "Invalid API key" com a key no log.

### Monitoramento para Solo Dev

A estratégia de monitoramento proposta é **necessária e suficiente** para o MVP. Os princípios corretos são:

1. **Alertas, não dashboards.** Fabio não tem tempo para olhar Grafana. Tudo precisa ser push (Telegram/Slack).
2. **GCP nativo primeiro.** Cloud Monitoring, Logging e Trace são incluídos. Sentry free tier para errors. Zero custo adicional.
3. **Monitoramento de custos é obrigatório.** Com pay-per-use, um bug que dispara queries BigQuery infinitas pode custar R$ 10K em uma noite. Budget alerts são inegociáveis.

## Parte 3 — Recomendações

### R1. Implementar "Adapter Pattern" para multi-provider LLM

**Problema:** Suportar Claude, Gemini e OpenAI significa 3 SDKs com interfaces diferentes, formatos de prompt diferentes, modelos de token diferentes.

**Recomendação:** Criar uma interface `LLMProvider` com métodos unificados:

```python
class LLMProvider(ABC):
    async def generate_sql(self, prompt: str, context: dict) -> str
    async def generate_insight(self, data: dict, query: str) -> str
    async def validate_key(self, key: str) -> bool
    async def estimate_cost(self, tokens: int) -> float
```

Implementações: `ClaudeProvider`, `GeminiProvider`, `OpenAIProvider`. Seleção por tenant config.

**Impacto:** Facilita adicionar novos providers (Llama, Mistral) e padroniza error handling.

### R2. Implementar cache de queries frequentes

**Problema:** Se 5 usuários do mesmo tenant perguntam "faturamento do mês" no mesmo dia, são 5 chamadas LLM idênticas + 5 queries BigQuery.

**Recomendação:**
- Cache de SQL gerado: mesma pergunta (normalizada) + mesmo schema → reutiliza SQL (TTL: 1h)
- Cache de resultado: mesmo SQL + dados não mudaram → reutiliza resultado (TTL: 15 min, configurável por tenant)
- Armazenamento: Firestore (já na stack)
- Economia estimada: 30-50% de chamadas LLM e BigQuery queries

### R3. Implementar "SQL Dry-Run" antes de toda execução

**Problema:** BigQuery cobra por TB processado. Uma query mal-gerada pelo LLM pode scanear TB de dados, custando caro.

**Recomendação:** Antes de executar qualquer query, fazer `BigQuery dry-run` (grátis) que retorna:
- Bytes que serão processados
- Se o SQL é válido
- Estimativa de custo

Se `estimated_bytes > threshold` (ex: 10GB), alertar o usuário: "Esta consulta vai processar muitos dados. Deseja continuar?" Isso protege tanto o custo do tenant quanto o custo da mAInd Tech (BigQuery quota).

### R4. Cloud SQL com HA desde o início

**Problema:** Cloud SQL sem HA é single point of failure. Se a instância cai, toda a plataforma para (metadata, configurações, glossários estão lá).

**Recomendação:** Habilitar HA (High Availability) com failover automático desde o MVP. Custo adicional: ~R$ 150-250/mês. Para uma plataforma que cobra R$ 497/mês por tenant, é custo aceitável a partir do 1o cliente Pro.

**Alternativa:** Se custo for restrição absoluta, usar Firestore para tudo (elimina Cloud SQL). Firestore é 99.999% disponível, serverless, e grátis até 1M ops/dia. Trade-off: perde SQL para queries analíticas de metadata.

### R5. Estratégia de monitoramento recomendada para solo dev

**Problema:** Com 1 dev, não há on-call. Incidentes fora do horário comercial dependem de Fabio ver o alerta.

**Recomendação — 3 camadas de defesa:**

| Camada | O que faz | Quando | Custo |
|--------|----------|--------|------:|
| **1. Auto-healing** | Cloud Run restart automático, circuit breakers, retries com backoff | Automático (24/7) | R$ 0 |
| **2. Alertas inteligentes** | Apenas alertas acionáveis (não noise). Telegram para SEV-1, email para SEV-2/3 | Quando threshold atingido | R$ 0 |
| **3. Status page** | Página pública de status (Instatus free tier ou GitHub Pages) | Contínuo | R$ 0 |

**Regras de alerta:**

| Tipo | Condição | Canal | Ação |
