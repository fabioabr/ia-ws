## 3. Metodologia de Trabalho

### 3.1 Framework: Scrum Adaptado

| Cerimônia | Frequência | Duração | Participantes | Objetivo |
|-----------|------------|---------|---------------|----------|
| **Daily Standup** | Diária (seg-sex) | 15 min | Todo o time | Sincronização, impedimentos, foco do dia |
| **Sprint Planning** | Quinzenal (início da sprint) | 2h | PO, Tech Lead, Devs, Designer | Selecionar itens do backlog, definir sprint goal |
| **Sprint Review** | Quinzenal (final da sprint) | 1h | Todo o time + CTO | Demo do incremento, feedback do sponsor |
| **Sprint Retro** | Quinzenal (após review) | 1h | Todo o time | Melhoria contínua do processo |
| **Backlog Refinement** | Semanal (meio da sprint) | 1h | PO, Tech Lead, dev representante | Preparar itens para a próxima sprint |
| **Technical Sync** | Semanal | 30 min | Tech Lead, Devs | Alinhamento técnico, decisões de arquitetura |

[INFERENCE]

### 3.2 Cadências e Sprints

| Parâmetro | Valor | Fonte |
|-----------|-------|-------|
| **Duração da sprint** | 2 semanas | [INFERENCE] |
| **Total de sprints no MVP** | 6 sprints | [BRIEFING][INFERENCE] |
| **Definition of Done** | Code reviewed, testes passando (≥80% coverage), deploy em staging, aceite do PO | [INFERENCE] |

## 5. Plano de Plantão (On-call)

### 5.1 Estratégia de On-call para MVP

| Aspecto | Detalhe | Fonte |
|---------|---------|-------|
| **Modelo** | Rotação semanal entre Tech Lead e Backend Devs (3 pessoas no pool) | [INFERENCE] |
| **Cobertura** | Horário comercial estendido: 08:00-22:00 BRT (seg-sex). Fora do horário: alertas críticos apenas. | [INFERENCE] |
| **Ferramenta de alertas** | Google Cloud Monitoring → PagerDuty (ou Opsgenie) → Slack | [INFERENCE] |
| **SLA de resposta** | P1 (sistema fora): 15 min; P2 (feature degradada): 1h; P3 (bug não-crítico): próximo dia útil | [INFERENCE] |
| **Escalação** | On-call → Tech Lead → CTO (apenas para P1 que requer decisão de negócio) | [INFERENCE] |

### 5.2 Classificação de Incidentes

| Severidade | Definição | Exemplos | SLA resposta | SLA resolução |
|-----------|-----------|----------|-------------|---------------|
| **P1 — Crítico** | Sistema completamente indisponível ou vazamento de dados | API down, cross-tenant data leak, billing cobrando errado | 15 min | 2h |
| **P2 — Alto** | Feature core degradada mas sistema funcional | Queries retornando erros intermitentes, latência > 30s, gráficos não renderizando | 1h | 8h |
| **P3 — Médio** | Feature não-core com problema | Exportação PDF falhando, sugestão de prompts não aparecendo | Próximo dia útil | 3 dias |
| **P4 — Baixo** | Cosmético ou melhoria | Typo na interface, cor errada no gráfico | Backlog | Próxima sprint |

[INFERENCE]

### 5.3 Runbooks Obrigatórios (a criar durante MVP)

| Runbook | Quando usar | Owner |
|---------|-------------|-------|
| API down — Cloud Run | Serviço não responde ou erro 5xx generalizado | Tech Lead |
| BigQuery timeout | Queries excedendo limites de tempo ou cota | Backend AI/ML |
| Cross-tenant data leak | Suspeita de acesso indevido entre tenants | Backend Plataforma |
| LLM API failure | Claude/Gemini API indisponível ou retornando erros | Backend AI/ML |
| Billing inconsistency | Divergência entre consumo real e cobrança | Backend Plataforma |
| Certificado expirado / Auth failure | Falha na autenticação generalizada | Tech Lead |

[INFERENCE]

## 6. Gestão de Mudança

### 6.1 Contexto

O Veezoozin é greenfield, então não há sistemas legados a migrar. A gestão de mudança foca em três eixos: [BRIEFING][INFERENCE]

1. **Mudança no time** — Construir uma equipe do zero, estabelecer cultura e processos
2. **Mudança no produto** — Iterações rápidas no MVP baseadas em feedback de tenants piloto
3. **Mudança no cliente** — Onboarding dos primeiros tenants, educação sobre o produto

### 6.2 Plano de Gestão de Mudança

#### Eixo 1 — Construção do Time

| Ação | Quando | Responsável | Fonte |
|------|--------|-------------|-------|
| Definir job descriptions e publicar vagas | Semana 0 | CTO | [INFERENCE] |
| Contratar Tech Lead e Backend AI/ML (prioridade 1) | Semana 1-2 | CTO | [INFERENCE] |
| Onboarding técnico: setup local, acesso GCP, arquitetura overview | Dia 1 de cada contratação | Tech Lead | [INFERENCE] |
| Estabelecer Definition of Done, coding standards, ADR template | Sprint 1 | Tech Lead + PO | [INFERENCE] |
| Primeira retrospectiva de time (ajuste de processo) | Final Sprint 1 | PO | [INFERENCE] |
| Avaliação de velocidade e ajuste de escopo se necessário | Final Sprint 2 | PO + CTO | [INFERENCE] |

#### Eixo 2 — Iteração de Produto

| Ação | Quando | Responsável | Fonte |
|------|--------|-------------|-------|
| Feature flags para deploys incrementais | A partir de Sprint 2 | Tech Lead | [INFERENCE] |
| Canary deploys (10% → 50% → 100%) para features críticas | A partir de Sprint 3 | Tech Lead | [INFERENCE] |
| Ciclo quinzenal de feedback com tenants piloto | A partir de Sprint 4 | PO | [INFERENCE] |
| Rollback plan documentado para cada deploy | Todo deploy | On-call | [INFERENCE] |
| Post-mortem para qualquer incidente P1 ou P2 | Até 48h após resolução | Tech Lead | [INFERENCE] |

#### Eixo 3 — Onboarding de Clientes

| Ação | Quando | Responsável | Fonte |
|------|--------|-------------|-------|
| Criar guia de onboarding self-service (Starter) | Sprint 5 | PO + Designer | [INFERENCE] |
| Criar playbook de onboarding assistido (Pro/Enterprise) | Sprint 5-6 | PO | [INFERENCE] |
| Recrutar 3-5 tenants piloto para beta fechado | Sprint 4-5 | CTO + PO | [INFERENCE] |
| Coletar feedback estruturado dos piloto (NPS + entrevista) | Quinzenal durante beta | PO | [INFERENCE] |
| Documentar FAQ e troubleshooting para suporte | Sprint 6 | PO + Tech Lead | [INFERENCE] |

### 6.3 Riscos de Mudança

| Risco | Impacto | Mitigação | Fonte |
|-------|---------|-----------|-------|
| Atraso na contratação de pessoas-chave (Tech Lead, Backend AI/ML) | Alto — atrasa todo o MVP | Pipeline de candidatos paralelo; contingência com freelancers seniores; CTO assume Tech Lead temporariamente | [INFERENCE] |
| Time novo sem coesão — velocity baixa nas primeiras sprints | Médio — atrasa entregas | Pair programming intensivo nas primeiras 2 sprints; mob programming para decisões de arquitetura | [INFERENCE] |
| Escopo creep durante o MVP | Alto — compromete prazo de 12 semanas | PO como gatekeeper rígido; qualquer adição de escopo exige remoção equivalente; CTO aprova mudanças de escopo | [INFERENCE] |
| Tenants piloto sem engajamento suficiente para feedback | Médio — MVP lançado sem validação real | Incentivos (plano Pro grátis por 6 meses); onboarding hands-on; canal direto com PO | [INFERENCE] |
| Dependência excessiva do CTO como único decisor | Médio — bottleneck | PO com autonomia para decisões de produto; Tech Lead com autonomia para decisões técnicas; CTO só para orçamento e estratégia | [BRIEFING][INFERENCE] |

## 8. Resumo Executivo

O Veezoozin será desenvolvido por um time enxuto de **7 pessoas** (incluindo CTO como sponsor parcial), usando **Scrum com sprints de 2 semanas** ao longo de **12 semanas (6 sprints)**. [BRIEFING][INFERENCE]

**Pontos-chave:**

- **Time crítico:** Tech Lead + Backend AI/ML são as primeiras contratações — sem eles, o projeto não inicia [INFERENCE]
- **Metodologia:** Scrum adaptado para startup — cerimônias leves, foco em entrega incremental [INFERENCE]
- **Governança:** CTO como decisor final (orçamento e estratégia), PO com autonomia de produto, Tech Lead com autonomia técnica [BRIEFING][INFERENCE]
- **On-call:** Rotação entre 3 devs, cobertura estendida (08-22h BRT), runbooks obrigatórios para cenários críticos [INFERENCE]
- **Gestão de mudança:** Foco em contratar rápido, estabelecer processos na Sprint 1 e recrutar tenants piloto até a Sprint 4-5 [INFERENCE]
- **Maior risco:** Atraso na contratação pode comprometer o prazo de 12 semanas — contingência com freelancers seniores [INFERENCE]
