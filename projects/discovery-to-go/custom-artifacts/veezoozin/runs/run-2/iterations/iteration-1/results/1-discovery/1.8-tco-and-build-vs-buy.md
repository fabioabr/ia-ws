## 2. Build vs Buy — Análise por Componente

> Referência: Bloco #5 (decisões de stack), Bloco #7 (módulos). Critérios: time-to-market (D17 — time enxuto), custo em 3 anos, risco operacional, diferenciação competitiva.

### 2.1. NL-to-SQL Engine

| Critério | Build (Custom) | Buy (Solução pronta) |
|----------|:--------------:|:--------------------:|
| **Opções** | LangChain + custom pipeline (D26) | Vanna.ai, Text2SQL (open-source), Dataherald |
| **Custo 3 anos** | ~R$ 0 (open-source + custom dev) | Vanna.ai: ~US$ 500/mês; Dataherald: ~US$ 2.000/mês |
| **Time-to-market** | 4-6 semanas (core pipeline) | 1-2 semanas (integração) |
| **Customização** | Total — glossário, RLS, cache, multi-idioma, privacy guard | Limitada — glossário básico, sem RLS nativo, sem privacy guard |
| **Diferenciação** | Contexto por tenant + RLS + privacy guard = moat | Funcionalidade commoditizada — qualquer concorrente pode usar |
| **Risco** | Complexidade de manutenção do pipeline | Lock-in, dependência de roadmap terceiro, limitação de customização |
| **Veredicto** | ✅ **BUILD** | ❌ |

> **Justificativa:** O NL-to-SQL Engine é o core do produto e o principal diferencial competitivo (Bloco #1 — moat). Comprar commoditiza o valor. O pipeline de 8 etapas (Bloco #5/7) com glossário, RLS, cache, privacy guard não existe pronto no mercado. O investimento em build é justificado pelo moat.

### 2.2. Vector Store (Embeddings)

| Critério | Build (Self-hosted) | Buy (Managed) |
|----------|:-------------------:|:-------------:|
| **Opções** | Qdrant/Weaviate self-hosted no GKE | Vertex AI Vector Search (GCP) / Pinecone |
| **Custo 3 anos** | Qdrant: ~R$ 120.000 (infra GKE) | Vertex AI: ~R$ 180.000 (managed) / Pinecone: ~R$ 250.000 |
| **Time-to-market** | 2-3 semanas (deploy + configuração) | < 1 semana (API pronta) |
| **Operação** | Requer SRE (backup, scaling, monitoring) | Zero operação — managed |
| **Integração GCP** | Manual (API custom) | Nativa — IAM, logging, monitoring integrados |
| **Risco** | Mais um componente para operar com time enxuto (D17) | Custo maior, mas operação trivial |
| **Veredicto** | ❌ | ✅ **BUY (Vertex AI Vector Search)** |

> **Justificativa:** Vector store não é diferencial competitivo. É infraestrutura. Time de 6 pessoas (D17) não deve operar Qdrant/Weaviate. Vertex AI Vector Search é GCP-native (créditos aplicáveis), zero operação, integração com Cloud Monitoring (D20). Custo maior justificado pela redução de carga operacional.

### 2.3. LLM (Modelo de Linguagem)

| Critério | Build (Fine-tune/Host) | Buy (API) |
|----------|:----------------------:|:---------:|
| **Opções** | Fine-tune Llama 3 no Vertex AI | Claude API (Anthropic) + Gemini API (Google) |
| **Custo 3 anos** | ~R$ 800.000 (GPU + treinamento + operação) | ~R$ 1.464.000 (API calls — seção 1.2) |
| **Time-to-market** | 3-6 meses (coleta dados, fine-tune, serving) | 1 dia (integração de API) |
| **Qualidade SQL gen** | Inferior no início (pouco dados), melhora com tempo | Estado da arte desde o dia 1 |
| **Operação** | ML Engineer dedicado, GPU, monitoring | Zero operação — API call |
| **Compliance** | Dados ficam 100% internos | Dados transitam por APIs externas (mitigado por D35, D39) |
| **Risco** | Qualidade inferior, custo de GPU, time de ML necessário | Dependência de provider, custo variável, privacidade |
| **Veredicto** | ❌ | ✅ **BUY (Claude + Gemini APIs)** |

> **Justificativa:** D3 — decisão do briefing. Sem time de ML (D17). Custo de API é ~83% maior em 3 anos, mas time-to-market é incomparável (1 dia vs 6 meses). Qualidade das APIs é superior. Cache (D46) reduz custo em 45-50% em escala. Mitigações de privacidade definidas em D35 e D39.

### 2.4. Autenticação

| Critério | Build (Custom) | Buy (Managed) |
|----------|:--------------:|:-------------:|
| **Opções** | Custom auth (bcrypt + JWT + sessões) | Firebase Auth / Auth0 / Clerk |
| **Custo 3 anos** | ~R$ 0 (dev time incluído na equipe) + ~R$ 50.000 (custo de manutenção/segurança) | Firebase: ~R$ 10.000 / Auth0: ~R$ 180.000 / Clerk: ~R$ 120.000 |
| **Time-to-market** | 3-4 semanas (incluindo 2FA, magic link) | < 1 semana |
| **SSO/SAML** | 4-8 semanas adicionais | Built-in (Firebase Auth + SAML provider) |
| **Segurança** | Risco de implementação insegura | Auditada, compliance SOC 2, OWASP |
| **Risco** | Vulnerabilidade de auth é catastrófica | Lock-in médio (migração possível) |
| **Veredicto** | ❌ | ✅ **BUY (Firebase Auth — D28)** |

> **Justificativa:** Auth é commoditizado e crítico para segurança. Implementação custom é risco desnecessário para time enxuto. Firebase Auth é GCP-native (D28), free tier generoso, suporte a SAML/OIDC para Enterprise na Fase 2. Persona Carla (Admin TI) exige segurança comprovada.

| **Time-to-market** | 8-12 semanas | 1-2 semanas |

| **Customização** | Total | Alta (Recharts/Nivo são React-based, themeable) |
| **Risco** | Over-engineering, desvio de foco do core (NL-to-SQL) | Limitações de tipos de gráfico avançados |
| **Veredicto** | ❌ | ✅ **BUY (Recharts/Nivo — open-source)** |

> **Justificativa:** Visualização é importante mas não é o moat (o moat é contexto + NL-to-SQL). Recharts/Nivo são React-based (D25 — Next.js), open-source, altamente customizáveis. Investir em rendering engine custom é desvio de foco. Se necessário, trocar de biblioteca no futuro é trivial.

### 2.6. Billing

| Critério | Build (Custom) | Buy (Plataforma) |
|----------|:--------------:|:----------------:|
| **Opções** | Custom billing engine | Stripe / Lago (open-source) / Chargebee |
| **Custo 3 anos** | ~R$ 150.000 (dev) + R$ 0 (fee) | Stripe: ~R$ 340.000 (fee 2.9% sobre receita projetada) / Lago: ~R$ 50.000 (hosting) |
| **Time-to-market** | 6-10 semanas (assinatura + usage + dunning + invoicing) | < 1 semana (Stripe) |
| **Assinatura + Usage** | Complexo de implementar corretamente | Nativo no Stripe (Billing + Usage Records) |
| **Dunning** | Implementar do zero | Built-in (smart retry, automático) |
| **PCI Compliance** | Precisa certificar (caro) | Stripe é PCI Level 1 (mais alto) |
| **Risco** | Bugs de billing = perda de receita + confiança | Fee contínuo, lock-in médio |
| **Veredicto** | ❌ | ✅ **BUY (Stripe — D27)** |

> **Justificativa:** D27 — antipattern #2 do blueprint SaaS: "billing custom no MVP é armadilha". Stripe é time-to-market < 1 semana. O fee de 2.9% sobre receita projetada totaliza ~R$ 340.000 em 3 anos, mas evita 6-10 semanas de desenvolvimento + PCI compliance + manutenção contínua. Lago (open-source) é alternativa se fee do Stripe se tornar pesado na escala.

### 2.7. Matriz Consolidada Build vs Buy

| Componente | Decisão | Custo 3 anos (R$) | Time-to-market | Justificativa principal |
|-----------|:-------:|:------------------:|:--------------:|-------------------------|
| **NL-to-SQL Engine** | 🔨 BUILD | ~R$ 0 (dev time na equipe) | 4-6 sem | Core product, moat competitivo |
| **Vector Store** | 🛒 BUY (Vertex AI) | ~R$ 180.000 | < 1 sem | Infra, não diferencial. GCP-native. |
| **LLM** | 🛒 BUY (Claude + Gemini) | ~R$ 1.464.000 | 1 dia | D3. Qualidade + time-to-market. |
| **Auth** | 🛒 BUY (Firebase) | ~R$ 10.000 | < 1 sem | Segurança crítica, GCP-native. |
| **Visualização** | 🛒 BUY (Recharts/Nivo) | R$ 0 (OSS) | 1-2 sem | Não é moat. Open-source. |
| **Billing** | 🛒 BUY (Stripe) | ~R$ 340.000 | < 1 sem | Antipattern #2. PCI. Dunning. |

> **Padrão:** Build apenas o que é diferencial competitivo (NL-to-SQL Engine). Buy tudo que é commodity (auth, billing, infra, visualização). Consistente com D3, D17 (time enxuto), D27, D28.

## 7. Recomendação Estratégica

### 7.1. Diagnóstico

O Veezoozin no cenário base **não atinge break-even em 3 anos**. Isso é **esperado para um SaaS B2B greenfield** — a maioria dos SaaS leva 3-5 anos para atingir breakeven total (incluindo investimento acumulado). O ponto crítico é:

1. **Equipe é 66,5% do TCO.** Otimizar equipe tem 4x mais impacto que otimizar infra.
2. **Enterprise é 60-70% da receita.** Cada tenant Enterprise vale ~6x um Pro. Foco em Enterprise é o caminho mais rápido para viabilidade.
3. **LLM APIs são o segundo maior custo variável.** Cache (D46) e migração gradual para Gemini (mais barato, créditos GCP) são alavancas reais.
4. **Break-even mensal é atingível.** No cenário D (pricing + crescimento), MRR > custos operacionais mensais no mês ~24. Isso significa que a operação se sustenta — o investimento acumulado é que leva mais tempo para recuperar.

### 7.2. Plano de ação recomendado

| Ação | Impacto | Prazo | Referência |
|------|---------|:-----:|-----------|
| **Lançar com pricing Pro R$ 697, Enterprise R$ 2.997** | +40-50% receita | Lançamento | Cenário A |
| **Foco em Enterprise desde o mês 6** (contratar SDR antes do planejado) | +60% receita Ano 2 | Mês 6 | Cenário C |
| **Cache agressivo desde o MVP** | -35% custo LLM | MVP | D30, D46 |
| **Migrar 40% do tráfego para Gemini até mês 12** | -20% custo LLM | Mês 12 | D26 |
| **Negociar créditos GCP agressivamente** | -30% infra GCP | Imediato | Briefing |
| **Revisar headcount trimestralmente** — não contratar por antecipação | -10-15% custo equipe | Contínuo | D17 |
| **Validar pricing nos 3 primeiros meses** — teste A/B | Confirmar ou ajustar | Mês 5-8 | Bloco #3 risk |

### 7.3. Métricas de gatilho (Go/No-Go)

| Marco | Métrica | Mínimo aceitável | Ação se não atingir |
|-------|---------|:-----------------:|---------------------|
| **Mês 8** (3 meses pós-lançamento) | Tenants pagantes | ≥ 8 Pro + 1 Enterprise | Rever pricing, rever posicionamento, considerar pivot |
| **Mês 12** | MRR | ≥ R$ 30.000 | Rever equipe, considerar cenário B (redução de escopo) |
| **Mês 18** | MRR | ≥ R$ 80.000 | Se < R$ 50K: cenário B agressivo. Se R$ 50-80K: manter e otimizar. |
| **Mês 24** | Break-even mensal | MRR ≥ custos mensais | Se não: avaliar captação de investimento ou redução para equipe mínima |
