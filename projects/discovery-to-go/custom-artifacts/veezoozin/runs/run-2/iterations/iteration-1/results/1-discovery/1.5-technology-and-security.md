## 2. Pipeline NL-to-SQL

> Referência do blueprint AI/ML — Componente 2 (Desenvolvimento de Modelos) e Componente 3 (MLOps e Serving). Referência do blueprint Datalake — Componente 1 (Aquisição de Dados, aplicado ao contexto de query).

### Fluxo completo do pipeline

```
┌──────────────────────────────────────────────────────────────────────────┐
│                        PIPELINE NL-to-SQL                                │
│                                                                          │
│  ┌───────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌─────────┐ │
│  │ INPUT │──▶│ CONTEXT  │──▶│ SQL GEN  │──▶│ VALIDATE │──▶│ EXECUTE │ │
│  │       │   │ ASSEMBLY │   │          │   │ + GUARD  │   │         │ │
│  └───────┘   └──────────┘   └──────────┘   └──────────┘   └────┬────┘ │
│  Pergunta     Schema +       LLM gera       Validação          │      │
│  NL do        Glossário +    SQL via         de segurança       │      │
│  usuário      Histórico +    Claude/Gemini   + permissão        │      │
│  (PT/EN/ES)   Permissões                                       │      │
│                                                           ┌────▼────┐ │
│  ┌───────┐   ┌──────────┐   ┌──────────┐                │ BigQuery│ │
│  │OUTPUT │◀──│ INSIGHT  │◀──│ RESULT   │◀───────────────│ (R/O)  │ │
│  │       │   │ GEN      │   │ PROCESS  │                └─────────┘ │
│  └───────┘   └──────────┘   └──────────┘                            │
│  Gráfico +    LLM interpreta  Formata                               │
│  Insight +    resultados      tabela, detecta                       │
│  Sugestão     + tendências    tipo de gráfico                       │
└──────────────────────────────────────────────────────────────────────────┘
```

### Etapas detalhadas

| Etapa | Responsabilidade | Latência alvo | Componentes |
|-------|-----------------|:-------------:|-------------|
| **1. Input** | Receber pergunta NL, detectar idioma (PT/EN/ES), normalizar | < 100ms | FastAPI endpoint, language detection |
| **2. Context Assembly** | Montar contexto do tenant: schema do BigQuery, glossário, histórico de conversas, permissões do usuário (RLS) | < 500ms | Embeddings (Vertex AI) para semantic search no schema, Cloud SQL para glossário, Firestore para histórico |
| **3. SQL Generation** | Enviar prompt estruturado ao LLM com contexto montado. LLM gera query SQL. | < 2s | Claude API (primário) ou Gemini API (fallback). Prompt template versionado. |
| **4. Validate + Guard** | Validar SQL: (a) é SELECT puro (D4 — read-only), (b) não acessa tabelas/campos bloqueados (D9), (c) injeta filtros RLS, (d) timeout e LIMIT aplicados | < 200ms | SQL parser custom (sqlglot), permissão engine, RLS injector |
| **5. Execute** | Executar query validada no BigQuery do tenant via service account read-only | < 2s (p50) | BigQuery Job API, service account com permissão de leitura apenas |
| **6. Result Process** | Formatar resultado: detectar tipo de dados, escolher tipo de gráfico ideal, paginar se necessário | < 300ms | Heurísticas de visualização (temporal → linhas, categórico → barras, etc.) |
| **7. Insight Generation** | LLM analisa resultados e gera insight textual + sugestão de próxima pergunta (D10) | < 1.5s | Claude/Gemini API. Prompt separado do SQL gen. |
| **8. Output** | Retornar resposta composta: dados + gráfico + insight + sugestão | — | JSON para frontend renderizar |

**Latência total alvo:** < 5 segundos (p50) conforme KR2.3 do Bloco #2.

### Modelo de LLM — Decisões

| Decisão | Escolha | Justificativa |
|---------|---------|---------------|
| **Abordagem** | LLM via API (sem fine-tuning) | [BRIEFING] D3 — sem LLM próprio. RAG com contexto do tenant (glossário + schema) substitui fine-tuning. Antipattern #9 do blueprint AI/ML. |
| **Provider principal** | Claude (Anthropic) | [INFERENCE] Melhor qualidade em SQL generation e contexto longo (200K tokens). Schemas com 50+ tabelas cabem no contexto. |
| **Provider secundário** | Gemini (Google) | [INFERENCE] Fallback quando Claude estiver indisponível. A/B testing para comparar qualidade e custo. Créditos GCP reduzem custo. |
| **Prompt strategy** | Few-shot com exemplos do tenant | [INFERENCE] Incluir 3-5 exemplos de queries bem-sucedidas do tenant no prompt. Melhora precisão sem fine-tuning. |
| **Re-treinamento** | Não aplicável (APIs externas) | [BRIEFING] Sem modelo próprio. Embeddings de schema re-gerados quando schema muda. |
| **Métrica de sucesso** | > 85% queries corretas na 1a tentativa | [BRIEFING] KR1.2. Feedback loop: thumbs up/down → ajustar glossário e exemplos. |
| **Custo por query** | Estimado R$ 0,10-0,20 (2 chamadas LLM por query) | [INFERENCE] SQL gen (~1K tokens input + 200 output) + insight gen (~500 tokens input + 200 output). Meta: < R$ 0,08 em 12 meses via cache (KR5.2). |

### Cache de queries

| Tipo | Estratégia | TTL | Economia estimada |
|------|-----------|-----|:-----------------:|
| **Exact match** | Hash da pergunta normalizada + tenant_id | 24h | ~15% das queries |
| **Semantic match** | Embedding da pergunta → nearest neighbor em cache | 24h | ~25% das queries |
| **Schema cache** | Schema do BigQuery do tenant em memória | Até schema change | Evita re-leitura a cada query |
| **Result cache** | Resultado de query SQL (BigQuery result) | 1h (configurável) | Reduz custo BigQuery |

> **Economia projetada:** Com exact + semantic cache, ~30-40% das queries não precisam de chamada LLM. Impacto no custo LLM: de R$ 1.500/mês (MVP) para ~R$ 1.000/mês. Em escala (50 tenants): de R$ 5.000 para ~R$ 3.500/mês.

## 3. Multi-tenant Strategy

> Referência do blueprint SaaS — Componente 2 (Tenancy e Infraestrutura). Resolve gap G3 do Bloco #2.

### Decisão de isolamento

| Abordagem | Prós | Contras | Veredicto |
|-----------|------|---------|:---------:|
| Database-per-tenant | Isolamento máximo, fácil compliance, performance isolada | Custo alto (1 Cloud SQL por tenant), complexidade operacional, migrations N vezes | ❌ Inviável para Free/Pro |
| Schema-per-tenant | Bom isolamento, um banco com N schemas | Complexidade de migrations, limite de schemas em PostgreSQL, custo médio | ❌ Overhead para time enxuto |
| **Row-level (tenant_id)** | Custo baixo (1 banco), simples operacionalmente, migrations 1x | Risco de cross-tenant leak se filtro falhar, performance pode degradar com volume | ✅ **Escolhido para MVP** |
| **Database dedicado (Enterprise)** | Isolamento total para Enterprise | Custo adicional, precisa de provisioning automático | ✅ **Fase 2 — para Enterprise com SLA 99.9%** |

### Implementação row-level

```
┌─────────────────────────────────────────────┐
│              Cloud SQL (PostgreSQL)           │
│                                               │
│  ┌─────────────────────────────────────────┐ │
│  │ Tabela: tenants                          │ │
│  │ id | name | plan | config | created_at   │ │
│  ├─────────────────────────────────────────┤ │
│  │ Tabela: users                            │ │
│  │ id | tenant_id | email | role | ...      │ │
│  ├─────────────────────────────────────────┤ │
│  │ Tabela: glossary_terms                   │ │
│  │ id | tenant_id | term | definition | ... │ │
│  ├─────────────────────────────────────────┤ │
│  │ Tabela: query_history                    │ │
│  │ id | tenant_id | user_id | question | ...│ │
│  └─────────────────────────────────────────┘ │
│                                               │
│  🔒 TODA query inclui WHERE tenant_id = ?     │
│  🔒 Middleware injeta tenant_id do JWT         │
│  🔒 Sem exceção — auditoria verifica           │
└─────────────────────────────────────────────┘
```

**Proteções contra cross-tenant leak:**

| Proteção | Implementação | Camada |
|----------|---------------|--------|
| Middleware obrigatório | Toda request extrai `tenant_id` do JWT e injeta no contexto | API (FastAPI) |
| Query builder com tenant_id | ORM (SQLAlchemy) com filter automático por tenant_id | Backend |
| Testes automatizados | Testes de integração que tentam acessar dados de outro tenant | CI/CD |
| Auditoria periódica | Query no Cloud SQL para detectar queries sem filtro de tenant_id | Observabilidade |
| RLS no PostgreSQL | Row-level security policies como camada extra de proteção | Banco |

## 4. Autenticação e Autorização

> Referência do Bloco #2: persona Carla (Admin TI) tem poder de veto se segurança não estiver coberta. Gap G4: SSO/SAML.

### Autenticação

| Aspecto | MVP | Fase 2 (Enterprise) |
|---------|-----|---------------------|
| **Método** | Email + senha (bcrypt) + Magic link | SSO (SAML 2.0 / OIDC) |
| **2FA** | TOTP (Google Authenticator) — opcional no MVP, obrigatório no Enterprise | TOTP + SMS + hardware key |
| **Provider** | Firebase Auth | Firebase Auth + SAML provider (Okta, Azure AD, Google Workspace) |
| **JWT** | Access token (15 min) + Refresh token (7 dias) | Idem + session management com logout remoto |
| **Sessão** | Firestore (sessão ativa, device tracking) | Idem + compliance de sessão única por device (Enterprise) |

### Autorização (RBAC)

> Referência D9: controle de acesso por campo/registro como feature core.

```
Tenant
  └── Roles
        ├── Owner (1 por tenant — quem criou)
        ├── Admin (Lucas — configura glossário, schema, acesso)
        ├── Manager (Marina — consulta + export + sugestões avançadas)
        ├── Analyst (Rafael — consulta, sem config)
        └── Viewer (somente leitura de dashboards compartilhados)
```

| Nível de acesso | Implementação | Tier mínimo |
|----------------|---------------|-------------|
| **Tabela** | Whitelist de tabelas permitidas por role no Cloud SQL | Free |
| **Campo** | Mascaramento/ocultação de campos sensíveis (CPF, salário) por role | Pro |
| **Registro (RLS)** | Filtro automático injetado na query SQL gerada (ex: `WHERE region = user.region`) | Enterprise |

**Implicação no pipeline NL-to-SQL:** Na etapa 4 (Validate + Guard), o engine injeta filtros de RLS antes de executar a query. Exemplo:

```sql
-- Query gerada pelo LLM:
SELECT region, SUM(revenue) FROM sales GROUP BY region;

-- Após injeção de RLS (Marina só vê região Sul):
SELECT region, SUM(revenue) FROM sales WHERE region = 'Sul' GROUP BY region;
```

## 5. Segurança

### Segurança em camadas

| Camada | Controle | Detalhes |
|--------|----------|----------|
| **Transporte** | TLS 1.3 obrigatório | Cloud Run suporta nativamente. HSTS header. |
| **Autenticação** | JWT + 2FA | Ver seção 4. |
| **Autorização** | RBAC + RLS | Ver seção 4. |
| **SQL Injection** | Parser + sanitização | sqlglot valida SQL gerado pelo LLM. Whitelist de funções SQL. Sem concatenação de strings. |
| **Query sandboxing** | Timeout + LIMIT + read-only | D4: read-only obrigatório. Timeout de 30 seg. LIMIT 10.000 rows. Sem DDL/DML. |
| **Secrets** | Secret Manager (GCP) | API keys, credenciais de BigQuery dos tenants, chaves de criptografia. Rotação automática. |
| **Criptografia at-rest** | AES-256 (Google-managed) | Cloud SQL, Firestore, Cloud Storage: criptografia padrão do GCP. BYOK no Enterprise (Fase 2). |
| **Criptografia in-transit** | TLS 1.3 | Todas as comunicações internas e externas. |
| **WAF** | Cloud Armor | Proteção contra DDoS, rate limit global, geo-blocking se necessário. |
| **Vulnerability scanning** | Container scanning (Artifact Registry) | Scan automático de imagens Docker no CI/CD. Bloqueia deploy se vulnerabilidade crítica. |
| **Dependency scanning** | Dependabot (GitHub) | Alertas automáticos de CVEs em dependências Python e JS. |

### Segurança específica do NL-to-SQL

| Risco | Mitigação | Prioridade |
|-------|-----------|:----------:|
| **Prompt injection** | Separar prompt do sistema do input do usuário. Validar SQL gerado independentemente do prompt. Nunca executar SQL que não passou pelo parser. | 🔴 Alta |
| **SQL injection via LLM** | sqlglot parse → whitelist de funções → AST validation → nenhum DML/DDL → re-serialize SQL limpo | 🔴 Alta |
| **Data exfiltration** | RLS injection, LIMIT obrigatório, log de toda query executada, alerta se query retorna > X rows | 🔴 Alta |
| **Credential theft** | Service accounts com permissão mínima (BigQuery Data Viewer). Secret Manager. Rotação de keys. | 🟡 Média |
| **LLM data leakage** | Não enviar dados do resultado ao LLM para insight gen se classificados como PII. Truncar resultados no prompt. | 🟡 Média |

## 6. Rate Limiting

> Referência D14: fair use de 5.000 queries/mês no Pro.

| Dimensão | Free | Pro | Enterprise |
|----------|:----:|:---:|:----------:|
| **Queries/dia** | 10 | Ilimitado (fair use ~167/dia) | Ilimitado |
| **Queries/minuto** | 2 | 20 | 60 |
| **Concurrent queries** | 1 | 3 | 10 |
| **Max result rows** | 1.000 | 10.000 | 100.000 |
| **Export/dia** | 0 | 10 | Ilimitado |
| **API calls/min (integrações)** | — | 30 | 120 |

**Implementação:** Redis counter (Memorystore na Fase 2). No MVP: contador em Firestore com TTL.

**Resposta ao atingir limite:**
- HTTP 429 com header `Retry-After`
- Mensagem amigável no frontend: "Você atingiu o limite de queries do seu plano. Upgrade para Pro para queries ilimitadas."
- Log de evento para analytics de churn/upgrade

## 9. Escalabilidade

> Referência do blueprint SaaS — Componente 2: "O que acontece quando o tenant mais pesado atinge 10x o volume atual?"

### Projeção de carga

| Cenário | Tenants | Usuários | Queries/dia | Infra |
|---------|:-------:|:--------:|:-----------:|-------|
| **MVP** | 12 | ~50 | ~300 | Cloud Run: 1-2 instâncias |
| **6 meses** | ~100 | ~500 | ~3.000 | Cloud Run: 5-10 instâncias, Memorystore |
| **12 meses** | ~300 | ~2.000 | ~15.000 | Cloud Run: 10-20 instâncias, database read replicas |

### Estratégia de scaling

| Componente | Estratégia | Trigger |
|-----------|-----------|---------|
| **Cloud Run (API)** | Auto-scaling horizontal | CPU > 70% ou requests/instance > 80 |
| **Cloud Run (Workers)** | Auto-scaling horizontal | Pub/Sub queue depth > 100 |
| **Cloud SQL** | Vertical (scale up) + read replicas (Fase 2) | CPU > 80% ou connections > 500 |
| **Firestore** | Automático (serverless) | Nenhum — escala transparente |
| **BigQuery** | Automático (serverless) | Nenhum — cobrança por bytes scanned |
| **LLM APIs** | Rate limit do provider | Distribuir entre Claude e Gemini |

### Hot tenant protection

> Um tenant Enterprise fazendo 60 queries/minuto não pode degradar a experiência dos outros.

| Proteção | Implementação |
|----------|---------------|
| **Rate limit por tenant** | Redis/Firestore counter por tenant_id (seção 6) |
| **Queue isolation** | Queries Enterprise em fila separada (Cloud Tasks) — Fase 2 |
| **Connection pooling** | PgBouncer para Cloud SQL — evita esgotamento de connections |
| **Circuit breaker** | Se BigQuery do tenant retorna erros > 50%, circuit breaker abre e retorna mensagem amigável |

## 10. Riscos Identificados neste Bloco

| Risco | Probabilidade | Impacto | Mitigação sugerida |
|-------|:------------:|:-------:|---------------------|
| Prompt injection no NL-to-SQL engine | Média | Alto | sqlglot parser, AST validation, separação de prompt do sistema vs user input. Nunca executar SQL sem validação. |
| Cross-tenant data leak via row-level | Baixa | Crítico | Middleware obrigatório, RLS no PostgreSQL, testes automatizados de isolamento, auditoria periódica. |
| Claude API indisponível por horas | Baixa | Alto | Gemini como fallback automático. Cache serve ~30-40% das queries sem LLM. Degradação graciosa. |
| Custo de LLM explode com escala | Média | Alto | Cache agressivo (exact + semantic), otimização de prompts (reduzir tokens), meta R$ 0,08/query em 12 meses. |
| Stripe rate limit ou indisponibilidade | Baixa | Médio | Webhook retry nativo. Jobs de reconciliação periódica. Sem impacto em queries (billing é assíncrono). |
| BigQuery do tenant sem permissão read-only | Média | Alto | Validação no onboarding: testar permissões da service account antes de aceitar conexão. Bloquear se tiver write. |
| Schema grande (500+ tabelas) excede contexto LLM | Baixa | Médio | Semantic search nos embeddings de schema — selecionar apenas tabelas relevantes para a pergunta. Não enviar schema inteiro. |
