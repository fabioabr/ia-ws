## 3. Jornadas do Usuário

### Jornada 1 — Primeiro Valor (Ativação)

> Referência do Bloco #1: time-to-value é crítico. O MVP deve entregar valor em minutos, não dias.

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  1. SIGNUP   │───▶│  2. CONECTAR │───▶│  3. MAPEAR   │───▶│ 4. PERGUNTAR│───▶│ 5. AHA!     │
│  Criar conta │    │  Banco de   │    │  Schema +    │    │ Primeira    │    │ Gráfico +   │
│  (Free/Pro)  │    │  dados      │    │  Glossário   │    │ pergunta NL │    │ insight     │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
     │                   │                  │                   │                  │
   Marina             Carla/Lucas         Lucas              Marina             Marina
   (self-service)     (técnico)           (domínio)          (NL)               (valor!)
     │                   │                  │                   │                  │
   < 2 min            < 10 min           < 30 min            < 30 seg           ATIVAÇÃO
```

**Time-to-value alvo:** < 45 minutos do signup até o primeiro gráfico com insight (inclui conexão do banco e mapeamento básico de schema).

**Momento "Aha!":** Marina faz a primeira pergunta em português ("Qual foi o faturamento por região no último trimestre?") e recebe um gráfico de barras com insight textual em menos de 5 segundos.

### Jornada 2 — Uso Diário (Retenção)

> Referência do Bloco #1: a persona primária (Marina) faz 2-5 consultas/dia. O sistema deve sugerir perguntas relevantes.

```
Marina abre o Veezoozin
    │
    ├──▶ Vê sugestões de prompts baseadas no contexto e histórico
    │       "Faturamento desta semana vs semana passada"
    │       "Top 10 clientes por receita no mês"
    │
    ├──▶ Faz pergunta em linguagem natural
    │       "Qual o churn por plano nos últimos 3 meses?"
    │
    ├──▶ Recebe resposta visual
    │       Gráfico de linhas (churn % por plano ao longo do tempo)
    │       Insight: "Churn do plano Basic subiu 23% no último mês"
    │       Sugestão: "Quer ver o breakdown por motivo de cancelamento?"
    │
    └──▶ Exporta para reunião (PDF/HTML)
```

**Métricas de sucesso da jornada:**
- Consultas por usuário/dia: alvo > 3
- Taxa de aceitação de sugestões: alvo > 40%
- Queries corretas na 1a tentativa: alvo > 85%
- Tempo de resposta: < 5 segundos para queries simples

### Jornada 3 — Onboarding do Tenant (Setup)

> Referência do Bloco #1: o glossário por tenant é o diferencial competitivo principal. O onboarding precisa ser eficiente.

```
Lucas (Admin Tenant) inicia o onboarding
    │
    ├──▶ 1. Conectar fonte de dados (BigQuery no MVP)
    │       → Carla (TI) valida credenciais e permissões (read-only)
    │       → Sistema testa conexão automaticamente
    │
    ├──▶ 2. Mapeamento automático de schema
    │       → Sistema descobre tabelas, colunas, tipos, relações
    │       → Gera descrições semânticas iniciais via LLM
    │       → Lucas revisa e ajusta ("esta coluna é receita bruta, não líquida")
    │
    ├──▶ 3. Glossário de negócio
    │       → Lucas define termos do domínio:
    │         "churn" = "cliente que cancelou nos últimos 30 dias"
    │         "cliente ativo" = "cliente com ao menos 1 transação nos últimos 90 dias"
    │       → Sistema incorpora no contexto das queries
    │
    ├──▶ 4. Configurar controle de acesso (dados sensíveis)
    │       → Marcar campos sensíveis (CPF, salário, etc.)
    │       → Definir quem pode ver quais dados (nível de registro/campo)
    │
    └──▶ 5. Validação
            → Lucas faz 5-10 perguntas de teste
            → Verifica se as respostas fazem sentido no contexto do negócio
            → Ajusta glossário conforme necessário
```

**Tempo alvo do onboarding completo:** < 2 horas para um banco com até 50 tabelas.

## 4. Estratégia de Onboarding por Tier

> Derivada do Bloco #1: modelo tiered (Free/Pro/Enterprise) implica onboarding diferenciado.

| Aspecto | Free | Pro | Enterprise |
|---------|------|-----|------------|
| **Tipo de onboarding** | 100% self-service | Self-service + docs + chat | White-glove (assistido) |
| **Setup do banco** | Wizard guiado, BigQuery apenas | Wizard + suporte técnico async | Engenheiro dedicado configura |
| **Glossário** | Templates prontos (genéricos) | Templates + customização livre | Consultoria de domínio (1-2h) |
| **Controle de acesso** | Básico (tudo-ou-nada) | Por campo/tabela | Por campo/registro + SSO + RBAC |
| **Time-to-value alvo** | < 30 min | < 2 horas | < 1 dia (com suporte) |
| **Limite de tabelas** | 5 | 50 | Ilimitado |

## 6. Controle de Acesso a Dados (Concern Especial)

> Referência do Bloco #1: o briefing destaca explicitamente a necessidade de controle de acesso em nível de registro/campo.

### Modelo de acesso proposto

```
Tenant
  └── Roles (Admin, Gestor, Analista, Viewer)
        └── Permissões por dataset/tabela
              └── Restrições por campo (ex: ocultar CPF para Viewers)
                    └── Restrições por registro (ex: região = "Sul" apenas)
```

| Nível | Descrição | Exemplo | Tier mínimo |
|-------|-----------|---------|-------------|
| **Tabela** | Acesso ou bloqueio a tabela inteira | "Analistas não acessam tabela de salários" | Free |
| **Campo** | Campos específicos ocultos ou mascarados | "CPF mascarado para Viewers" | Pro |
| **Registro** | Filtro automático por atributo do usuário | "Gestor regional vê apenas dados da sua região" | Enterprise |

**Implicação técnica:** O controle por registro exige injeção de filtros na query SQL gerada antes da execução. O engine de NL-to-SQL deve suportar "row-level security" como concern nativo, não como afterthought.
