# Bloco #6 — Privacidade e Compliance

> Resultado da entrevista simulada (Fase 1 — Discovery) entre Cyber Security Architect e Customer simulado. Este bloco cobre LGPD em modo profundo — o Veezoozin processa queries que podem conter e retornar dados pessoais (PII) dos clientes dos tenants. Roda em **paralelo** com o Bloco #5 (Tecnologia e Segurança) conforme decisão D23.
>
> **Por que modo profundo?** O Veezoozin é um SaaS multi-tenant que: (a) coleta dados pessoais de usuários no signup, (b) processa queries sobre bancos que contêm PII dos clientes do tenant, (c) envia perguntas em linguagem natural para APIs de LLM externas (Claude, Gemini), (d) gera insights que podem revelar informações sobre pessoas. O blueprint SaaS confirma: "em SaaS o modo profundo é quase sempre o caso".
>
> Referencia: personas do Bloco #2 (especialmente Carla, Admin TI — poder de veto em segurança), D4 (read-only), D9 (controle de acesso campo/registro), D24 (row-level multi-tenancy).

## 1. Classificação de Dados

### Inventário de dados pessoais

| Categoria | Dados | Origem | Armazenamento | Sensibilidade |
|-----------|-------|--------|---------------|:-------------:|
| **Dados de conta** | Nome, email, telefone, cargo | Signup e perfil do usuário | Cloud SQL (metadata) | 🟡 Pessoal |
| **Dados de autenticação** | Hash de senha, tokens 2FA, sessões | Login | Firebase Auth + Firestore | 🔴 Sensível |
| **Dados de uso** | Queries feitas, horário, frequência, IP | Uso do produto | Cloud SQL + Firestore | 🟡 Pessoal |
| **Dados de billing** | Cartão (via Stripe — não armazenado), NF-e, endereço fiscal | Pagamento | Stripe (PCI compliant) + Cloud SQL (referência) | 🔴 Sensível |
| **Dados de query (input)** | Perguntas em linguagem natural que podem conter PII | Uso do produto | Firestore (histórico) | 🔴 Potencialmente sensível |
| **Dados de query (output)** | Resultados de queries SQL que podem conter PII do banco do tenant | Execução de query | Transiente (não persistido por padrão) | 🔴 Potencialmente sensível |
| **Dados de contexto** | Glossário, schema, embeddings | Configuração do tenant | Cloud SQL + Vertex AI | 🟢 Não pessoal (metadata) |
| **Dados do banco do tenant** | Qualquer dado no BigQuery do tenant (incluindo PII dos clientes do tenant) | BigQuery do tenant (read-only) | **Não armazenado pelo Veezoozin** — query results são transientes | 🔴 Potencialmente crítico |
| **Logs de auditoria** | Quem acessou o quê, quando, qual query | Operação | Cloud Logging | 🟡 Pessoal (contém user_id) |

### Classificação por nível

```
┌───────────────────────────────────────────────────────┐
│                  CLASSIFICAÇÃO DE DADOS                │
│                                                        │
│  🔴 CRÍTICO (dados do banco do tenant)                │
│     PII dos clientes do tenant — CPF, nome, salário,  │
│     saúde, etc. Veezoozin NÃO armazena, apenas        │
│     consulta e exibe. Mas transita pelo pipeline.      │
│                                                        │
│  🔴 SENSÍVEL (auth + billing)                         │
│     Credenciais, tokens, dados de pagamento.           │
│     Armazenados com criptografia. Stripe PCI.          │
│                                                        │
│  🟡 PESSOAL (conta + uso + logs)                      │
│     Identificável. Base legal necessária.              │
│     Sujeito a direitos do titular (LGPD art. 18).     │
│                                                        │
│  🟢 NÃO PESSOAL (contexto, metadata técnica)          │
│     Glossário, schema, embeddings, métricas agregadas. │
│     Sem PII. Livre tratamento.                         │
└───────────────────────────────────────────────────────┘
```

## 2. Papéis LGPD — Controlador vs Operador

> O Veezoozin opera em um modelo de camadas com papéis distintos conforme a LGPD.

### Modelo de responsabilidades

```
┌─────────────────────────────────────────────────────────┐
│                CLIENTES DO TENANT                        │
│  (pessoas cujos dados estão no BigQuery do tenant)      │
│  → São os TITULARES dos dados no banco                  │
└────────────────────────┬────────────────────────────────┘
                         │ dados pessoais
┌────────────────────────▼────────────────────────────────┐
│              EMPRESA CLIENTE (Tenant)                     │
│  → É o CONTROLADOR dos dados no banco                    │
│  → Define finalidade e meio de tratamento                │
│  → Decide quais dados o Veezoozin pode consultar         │
│  → Responsável por ter base legal para tratar os dados   │
└────────────────────────┬────────────────────────────────┘
                         │ contrato de serviço
┌────────────────────────▼────────────────────────────────┐
│              mAInd Tech (Veezoozin)                      │
│  → É o OPERADOR dos dados do banco do tenant             │
│  → Trata dados sob instrução do controlador (tenant)     │
│  → É CONTROLADOR dos dados de conta/uso dos usuários     │
│  → Precisa de DPA (Data Processing Agreement) com tenant │
└────────────────────────┬────────────────────────────────┘
                         │ chamadas API
┌────────────────────────▼────────────────────────────────┐
│              SUB-OPERADORES                              │
│  Claude (Anthropic), Gemini (Google), Stripe, SendGrid  │
│  → Tratam dados sob instrução do operador (mAInd Tech)  │
│  → Precisam de DPA com mAInd Tech                       │
└─────────────────────────────────────────────────────────┘
```

### Duplo papel da mAInd Tech

| Contexto | Papel LGPD | Base legal | Dados |
|----------|-----------|-----------|-------|
| Dados de conta e uso dos usuários do Veezoozin | **Controlador** | Execução de contrato (art. 7, V) + legítimo interesse (art. 7, IX) | Nome, email, queries feitas, histórico |
| Dados do banco do tenant (PII dos clientes do tenant) | **Operador** | Sob instrução do controlador (tenant) | Qualquer PII que transitar nas queries |

## 3. Sub-processadores (Sub-operadores)

> Referência do blueprint SaaS: "Sub-operadores (Stripe, SendGrid, Datadog) — contratos DPA".

### Lista de sub-processadores

| Sub-processador | Dados que recebe | Finalidade | Residência de dados | DPA disponível |
|-----------------|-----------------|-----------|:-------------------:|:--------------:|
| **Anthropic (Claude API)** | Perguntas NL do usuário + contexto do tenant (schema, glossário) + resultado de queries (para insight gen) | Geração de SQL e insights | 🇺🇸 EUA | ✅ Sim ([anthropic.com/privacy](https://anthropic.com)) |
| **Google (Gemini / Vertex AI)** | Perguntas NL + contexto (fallback) + texto para embeddings | LLM fallback, embeddings de schema | 🇧🇷 / 🇺🇸 Configurável | ✅ Sim (Google Cloud DPA) |
| **Stripe** | Nome, email, endereço fiscal, dados de pagamento (tokenizado) | Billing e processamento de pagamento | 🇺🇸 EUA (PCI compliant) | ✅ Sim (Stripe DPA) |
| **SendGrid / Resend** | Email do usuário + nome | Emails transacionais | 🇺🇸 EUA | ✅ Sim |
| **Google Cloud Platform** | Todos os dados (infra provider) | Hosting, storage, compute | 🇧🇷 (southamerica-east1) | ✅ Sim (Google Cloud DPA) |

### Riscos de PII em chamadas LLM

> Este é o concern mais crítico de privacidade do Veezoozin.

| Cenário | Risco | Mitigação |
|---------|-------|-----------|
| Pergunta do usuário contém PII | "Qual o salário do João Silva?" → PII enviada ao Claude/Gemini | **M1:** Documentar no DPA que queries contêm PII sob instrução do controlador (tenant). **M2:** Anthropic e Google não usam dados de API para treinamento (validar contratualmente). |
| Resultado de query contém PII e vai para insight gen | SELECT nome, cpf, salário → resultado enviado ao LLM para gerar insight | **M3:** Truncar resultados antes de enviar ao LLM — enviar apenas agregações, não dados individuais. **M4:** Campo marcado como "sensível" no glossário → nunca incluir valores individuais no prompt de insight. |
| Schema contém nomes de tabelas/colunas com PII implícita | Coluna "cpf", "rg", "salario" → revelam existência de PII | **M5:** Nomes de colunas não são PII em si (são metadata). Risco baixo. Documentar no RIPD. |
| Histórico de conversas contém PII | Perguntas anteriores do usuário mencionam nomes, CPFs | **M6:** TTL no histórico (7/90/365 dias por plano — D11). **M7:** Opção para tenant desabilitar histórico de conversas. |

### Controles obrigatórios para sub-processadores

| Controle | Implementação | Status |
|----------|---------------|:------:|
| DPA assinado com cada sub-processador | Verificar e assinar DPA com Anthropic, Google, Stripe, email provider | 🔶 A fazer |
| Verificar política de uso de dados para treinamento | Anthropic API e Google Vertex AI não usam dados de clientes para treinamento (confirmar) | 🔶 A verificar |
| Minimização de dados enviados ao LLM | Truncar resultados, enviar apenas schema relevante (não completo), não enviar PII individual | ✅ Design definido |
| Logs de dados enviados ao LLM | Registrar hash + tamanho de cada prompt enviado (sem conteúdo) para auditoria | ✅ Design definido |
| Direito de auditoria sobre sub-processadores | Cláusula no contrato com tenant permitindo auditoria do uso dos sub-processadores | 🔶 A incluir no contrato |

## 5. Bases Legais (LGPD art. 7)

### Por categoria de dados

| Categoria de dados | Base legal | Artigo LGPD | Justificativa |
|-------------------|-----------|:-----------:|---------------|
| **Dados de conta** (nome, email) | Execução de contrato | Art. 7, V | Necessários para prestar o serviço contratado |
| **Dados de autenticação** (hash senha, 2FA) | Execução de contrato | Art. 7, V | Necessários para garantir acesso seguro |
| **Dados de billing** (endereço fiscal, NF-e) | Obrigação legal | Art. 7, II | Nota fiscal é obrigação legal (legislação tributária) |
| **Dados de uso** (queries, frequência, IP) | Legítimo interesse | Art. 7, IX | Melhoria do serviço, detecção de anomalias, anti-fraude. Sujeito a balanceamento de interesses. |
| **Histórico de conversas** | Execução de contrato | Art. 7, V | Feature core do produto — histórico faz parte do serviço contratado |
| **Dados de query (banco do tenant)** | Instrução do controlador | Art. 7, V (via DPA) | mAInd Tech como operadora trata dados sob instrução do tenant (controlador) |
| **Dados enviados a LLMs** | Legítimo interesse + instrução do controlador | Art. 7, IX + DPA | Necessário para funcionalidade core. Minimização aplicada. |
| **Logs de auditoria** | Legítimo interesse + obrigação legal | Art. 7, IX + II | Segurança, compliance, auditoria |
| **Cookies e telemetria** | Consentimento | Art. 7, I | Cookie banner com opt-in para cookies não essenciais |

### Teste de legítimo interesse (LIA) — Dados de uso

| Critério | Avaliação |
|----------|-----------|
| **Finalidade legítima** | Melhoria do serviço (sugestão de prompts), detecção de anomalias (queries suspeitas), anti-fraude, métricas de produto |
| **Necessidade** | Sem dados de uso, impossível melhorar precisão de queries, detectar churn, ou sugerir prompts. Não há alternativa menos invasiva. |
| **Balanceamento** | Dados de uso são expectativa razoável do titular ao usar uma plataforma SaaS. Não inclui dados sensíveis do banco do tenant. Transparência via política de privacidade. |
| **Salvaguardas** | Pseudonimização em analytics (aggregation), TTL por plano, direito de opt-out de telemetria avançada. |
| **Conclusão** | ✅ Legítimo interesse aplicável com salvaguardas. |

## 7. Direito ao Esquecimento em Multi-tenant

> Referência do blueprint SaaS: "Tenant que solicita exclusão LGPD — como remover dados sem quebrar referências históricas?"

### Estratégia de exclusão

```
Solicitação de exclusão
    │
    ├── 1. SOFT-DELETE (imediato)
    │      → Marcar registros como deleted_at = NOW()
    │      → Dados não aparecem em queries
    │      → Tenant vê mensagem "conta em processo de exclusão"
    │      → Período de recuperação: 30 dias
    │
    ├── 2. ANONYMIZATION (D+30)
    │      → Substituir PII por valores genéricos:
    │        nome → "Usuário Anônimo #hash"
    │        email → "deleted_hash@anon.veezoozin.com"
    │        queries → manter SQL anônimo para analytics (sem PII)
    │      → Manter métricas agregadas (contagem de queries, etc.)
    │
    └── 3. HARD-DELETE (D+60)
           → Remover registros do Cloud SQL
           → Remover embeddings do Vertex AI
           → Remover sessões do Firestore
           → Remover logs com PII do Cloud Logging (set retention)
           → Confirmar exclusão via email ao solicitante
```

### Dados que NÃO são excluídos

| Dado | Motivo de retenção | Base legal |
|------|-------------------|-----------|
| Notas fiscais emitidas | Obrigação legal tributária (5 anos) | Art. 7, II |
| Logs de auditoria (anonimizados) | Obrigação legal de segurança + legítimo interesse | Art. 7, II + IX |
| Métricas agregadas (sem PII) | Não são dados pessoais | Não aplicável |

## 9. Transferência Internacional de Dados

> Referência: Claude API (Anthropic — EUA), Stripe (EUA), SendGrid (EUA). LGPD art. 33.

### Mapeamento de fluxos internacionais

| Fluxo | Dados | Destino | Base legal para transferência |
|-------|-------|:-------:|-------------------------------|
| Query NL → Claude API | Pergunta do usuário + contexto do tenant | 🇺🇸 EUA | Cláusulas contratuais padrão (SCC) no DPA com Anthropic |
| Query NL → Gemini API (Vertex) | Pergunta do usuário + contexto | 🇧🇷 BR (configurável) ou 🇺🇸 EUA | Se BR: sem transferência internacional. Se EUA: Google Cloud DPA + SCC. |
| Billing → Stripe | Dados de pagamento | 🇺🇸 EUA | SCC no DPA com Stripe. PCI-DSS compliance. |
| Email → SendGrid/Resend | Email + nome | 🇺🇸 EUA | SCC no DPA com email provider. |
| Infra → GCP | Todos os dados | 🇧🇷 BR (southamerica-east1) | **Sem transferência internacional** — região BR configurada. |

### Mitigações para transferência aos EUA

| Mitigação | Implementação |
|-----------|---------------|
| **Preferir processamento no Brasil** | GCP região southamerica-east1 para infra. Gemini via Vertex AI com endpoint regional BR quando disponível. |
| **SCC com todos os sub-processadores nos EUA** | Verificar e manter SCC atualizadas com Anthropic, Stripe, SendGrid. |
| **Minimização** | Enviar apenas o mínimo necessário ao LLM. Truncar resultados. Não enviar PII individual. |
| **Transparência** | Informar na política de privacidade que dados podem ser processados nos EUA para funcionalidades específicas. |
| **Opção Enterprise** | Para Enterprise com requisito de residência total no BR: usar apenas Gemini (Vertex AI BR) como LLM, sem Claude. Feature flag por tenant. |

## 10. RIPD — Relatório de Impacto à Proteção de Dados

> LGPD art. 38: a ANPD pode solicitar RIPD ao controlador.

### Necessidade

O Veezoozin **deve** ter RIPD porque:
1. Trata dados pessoais em larga escala (múltiplos tenants com dados de seus clientes)
2. Utiliza tecnologias de IA (LLMs) para processar dados que podem conter PII
3. Realiza transferência internacional de dados (APIs nos EUA)
4. Opera como plataforma multi-tenant com risco de cross-tenant leak

### Estrutura do RIPD (a ser elaborado)

| Seção | Conteúdo | Status |
|-------|----------|:------:|
| Descrição do tratamento | Pipeline NL-to-SQL, fluxo de dados, sub-processadores | 🔶 A elaborar |
| Necessidade e proporcionalidade | Por que LLM externo? Por que transferência internacional? | 🔶 A elaborar |
| Riscos ao titular | Cross-tenant leak, PII em prompts LLM, data breach | 🔶 A elaborar |
| Medidas de mitigação | RLS, criptografia, DPAs, minimização, cache, anonimização | ✅ Definidas neste bloco |
| Parecer do DPO | Recomendação de prosseguir com salvaguardas | 🔶 Após contratar DPO |

| **Read-only** | O Veezoozin apenas lê dados do banco do tenant. Nunca modifica, insere ou exclui dados no banco do cliente. (D4) |

| **Uso de LLMs externos** | Queries são processadas por APIs de IA (Claude, Gemini) para gerar SQL e insights. Dados são minimizados. Provedores não usam dados para treinamento. |
| **Isolamento multi-tenant** | Dados de um tenant nunca são acessíveis por outro tenant. Isolamento técnico descrito no DPA. |

## 14. Riscos Identificados neste Bloco

| Risco | Probabilidade | Impacto | Mitigação sugerida |
|-------|:------------:|:-------:|---------------------|
| PII transitando em chamadas LLM sem controle | Alta | Alto | Minimização de dados nos prompts (M3, M4). DPA com Anthropic/Google confirmando não uso para treinamento. Truncar resultados antes de insight gen. |
| Cross-tenant data leak via bug no row-level filtering | Baixa | Crítico | Middleware obrigatório, RLS no PostgreSQL, testes automatizados, auditoria periódica (D24). |

| ANPD solicita RIPD e não está pronto | Média | Alto | Elaborar RIPD antes do lançamento do MVP (semana 13-14). Incluir no cronograma. |
| Sub-processador muda política de dados | Baixa | Alto | Monitorar termos de serviço de Anthropic, Google, Stripe. Alertar DPO em caso de mudança. Ter fallback (Gemini se Anthropic mudar política). |
| Tenant não tem base legal para dados que disponibiliza | Média | Alto | Cláusula no contrato: tenant declara ter base legal. Veezoozin como operador não é responsável, mas deve documentar no DPA. |
| Dados de histórico não excluídos no prazo (bug) | Média | Médio | Firestore TTL automático + job de verificação diário + alerta se dados expirados não foram removidos. |
