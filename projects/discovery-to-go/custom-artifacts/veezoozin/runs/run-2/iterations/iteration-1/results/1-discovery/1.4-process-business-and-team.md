## 3. Metodologia de Desenvolvimento

### Framework

| Aspecto | Decisão | Justificativa |
|---------|---------|---------------|
| **Metodologia** | Scrum adaptado (sprints curtos) | Time pequeno precisa de ritmo. Scrum dá cadência sem overhead de kanban puro. |
| **Duração do sprint** | 1 semana | MVP de 4 meses = 16 sprints. Sprints curtos permitem pivots rápidos. |
| **Cerimônias** | Planning (seg), Daily (15 min), Review + Retro (sex) | Mínimo viável de cerimônias. Sem overhead. |
| **Ferramenta de gestão** | Linear ou GitHub Projects | Lightweight. Integração direta com GitHub. |
| **Repositório** | Monorepo (GitHub) | Time pequeno, produto único. Simplifica CI/CD e code review. |
| **Branching** | Trunk-based development com feature flags | Deploy contínuo, sem branches de longa vida. Feature flags controlam rollout. |

### Cadência de entregas

```
Semana 1-2:   Setup — Infra GCP, monorepo, CI/CD, auth básica
Semana 3-4:   Core — Engine NL-to-SQL básico (BigQuery), schema discovery
Semana 5-6:   Core — Glossário por tenant, contexto de negócio
Semana 7-8:   UX — Interface conversacional, gráficos automáticos
Semana 9-10:  Features — Sugestão de prompts, export, histórico
Semana 11-12: Polish — Onboarding wizard, billing (Stripe), planos
Semana 13-14: QA — Testes e2e, load testing, security review
Semana 15-16: Launch — Early adopters (5 tenants), monitoramento, ajustes
```

## 5. SLA e SLO/SLI

> Referência do Bloco #3: Free = best-effort, Pro = 99.5%, Enterprise = 99.9%.

### SLIs (Service Level Indicators)

| SLI | O que mede | Como mede |
|-----|-----------|-----------|
| **Disponibilidade** | % de requests HTTP com status 2xx/3xx | Uptime monitoring (externo) + health checks |
| **Latência de query** | Tempo entre envio da pergunta NL e resposta visual | p50, p90, p99 medidos no backend |
| **Precisão de query** | % de queries SQL geradas que retornam dados corretos | Feedback implícito (user aceita) + explícito (thumbs up/down) |
| **Tempo de onboarding** | Minutos entre signup e primeira query com resultado | Instrumentação no frontend |

### SLOs (Service Level Objectives)

| SLO | Free | Pro | Enterprise |
|-----|:----:|:---:|:----------:|
| Disponibilidade mensal | — (best-effort) | ≥ 99.5% (~3.6h downtime/mês) | ≥ 99.9% (~43 min downtime/mês) |
| Latência p50 (query simples) | < 8 seg | < 5 seg | < 3 seg |
| Latência p99 (query simples) | < 30 seg | < 15 seg | < 10 seg |
| Precisão de query | > 80% | > 85% | > 90% |
| Tempo de onboarding | < 30 min | < 2 horas | < 1 dia (com suporte) |

### SLA (contratual — somente Enterprise)

| Compromisso | Valor | Penalidade por descumprimento |
|-------------|:-----:|-------------------------------|
| Disponibilidade mensal | ≥ 99.9% | Crédito de 10% do MRR por cada 0.1% abaixo |
| Tempo de resposta a incidente crítico | < 1 hora | Crédito de 5% do MRR se exceder |
| Resolução de incidente crítico | < 4 horas | Menção no post-mortem público |

## 6. Observabilidade

> Referência do blueprint SaaS: observabilidade per tenant é concern do Componente 4.

### Stack de observabilidade

| Camada | Ferramenta | Justificativa |
|--------|-----------|---------------|
| **Métricas** | Google Cloud Monitoring + Prometheus | GCP-native, custo controlado, integração com Cloud Run |
| **Logs** | Google Cloud Logging (estruturado, JSON) | GCP-native, busca por tenant_id como label |
| **Traces** | Google Cloud Trace (OpenTelemetry) | Trace distribuído: request → NL engine → SQL → BigQuery → resposta |
| **Dashboards** | Grafana Cloud (free tier) ou Looker Studio | Dashboards por tenant, por plano, por feature |
| **Alertas** | Google Cloud Monitoring Alerting | PagerDuty para on-call (Fase 2). Slack no MVP. |
| **APM de negócio** | Custom (métricas no Firestore/BigQuery) | Queries/dia por tenant, precisão, churn signals |

### Dimensões obrigatórias em todos os logs/métricas

```json
{
  "tenant_id": "uuid",
  "plan": "free|pro|enterprise",
  "user_id": "uuid",
  "persona_type": "gestor|analista|admin_tenant|admin_ti",
  "request_type": "nl_query|export|config|auth",
  "latency_ms": 1234,
  "llm_provider": "claude|gemini",
  "llm_cost_usd": 0.003,
  "query_accuracy": "accepted|rejected|unknown"
}
```

> **Custo estimado de observabilidade:** R$ 300/mês no MVP (GCP-native), escala para ~R$ 3.000/mês com 100+ tenants. Consistente com projeção do Bloco #3.

## 7. On-Call e Resposta a Incidentes

### Modelo MVP (time pequeno)

| Aspecto | Definição |
|---------|-----------|
| **Modelo** | Rotação semanal entre Tech Lead + 2 Backend Engineers |
| **Horário** | Comercial (9h-18h) no MVP. 24/7 quando houver Enterprise com SLA. |
| **Canal de alerta** | Slack (#alerts-veezoozin) → telefone para P1 |
| **Tempo de resposta** | P1 (sistema down): < 15 min. P2 (degradado): < 1h. P3 (bug): próximo sprint. |
| **Runbook** | Mínimo: restart Cloud Run, verificar BigQuery quotas, verificar LLM APIs, verificar auth. |
| **Post-mortem** | Obrigatório para todo P1. Template: timeline, root cause, impact, action items. |

### Classificação de severidade

| Severidade | Descrição | Exemplo | SLA de resposta |
|:----------:|-----------|---------|:---------------:|
| **P1** | Sistema indisponível para todos os tenants | Cloud Run down, auth quebrada | < 15 min |
| **P2** | Funcionalidade degradada ou 1 tenant afetado | BigQuery timeout para tenant X, gráficos não renderizam | < 1h |
| **P3** | Bug não bloqueante | Tradução incorreta, sugestão irrelevante | Próximo sprint |
| **P4** | Melhoria / cosmético | Alinhamento de UI, mensagem de erro pouco clara | Backlog |

### Evolução do on-call

| Fase | Modelo | Justificativa |
|------|--------|---------------|
| MVP | Rotação semanal, horário comercial | Time pequeno, sem Enterprise ainda |
| Fase 2 (+SRE) | Rotação com SRE dedicado, horário estendido (8h-22h) | Primeiros Enterprise, SLA contratual |
| Fase 3 | SRE team 24/7 + PagerDuty | Escala exige cobertura total |

## 8. Feature Flags e Release Strategy

### Feature flags

| Aspecto | Decisão |
|---------|---------|
| **Ferramenta** | Custom simples no MVP (Firestore config por tenant). LaunchDarkly na Fase 2. |
| **Granularidade** | Por tenant, por plano, por % de usuários |
| **Casos de uso** | Rollout gradual de features, A/B testing de pricing, kill switch para features problemáticas |

### Estratégia de release

| Aspecto | MVP | Pós-MVP |
|---------|-----|---------|
| **Frequência de deploy** | Diário (trunk-based) | Diário (canary) |
| **Estratégia** | Rolling update (Cloud Run) | Canary → rolling (Cloud Run traffic splitting) |
| **Rollback** | Revert do deploy (Cloud Run revision) | Automático se error rate > 5% |
| **Feature freeze** | Últimas 2 semanas antes do launch | Antes de QBRs enterprise |

## 9. CI/CD Pipeline

```
┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐
│  Push   │───▶│  Lint + │───▶│  Unit   │───▶│  Build  │───▶│ Deploy  │
│  (main) │    │  Format │    │  Tests  │    │  Docker │    │ Staging │
└─────────┘    └─────────┘    └─────────┘    └─────────┘    └────┬────┘
                                                                  │
                                                            ┌─────▼─────┐
                                                            │ Integration│
                                                            │   Tests    │
                                                            └─────┬─────┘
                                                                  │
                                                            ┌─────▼─────┐
                                                            │  Deploy   │
                                                            │   Prod    │
                                                            │ (canary)  │
                                                            └───────────┘
```

| Etapa | Ferramenta | Critério de gate |
|-------|-----------|-----------------|
| Lint + Format | Ruff (Python), ESLint (TS) | Zero warnings |
| Unit Tests | pytest, Vitest | Cobertura > 80% |
| Build | Docker (Cloud Build) | Build sucesso |
| Deploy Staging | Cloud Run (staging) | Health check OK |
| Integration Tests | pytest + Playwright | Smoke tests passam |
| Deploy Prod | Cloud Run (prod, canary 10%) | Error rate < 1% por 10 min |
| Promote | Cloud Run (100% traffic) | Automático se canary OK |

## 10. Estratégia de Testes

| Tipo | Escopo | Ferramenta | Cobertura mínima |
|------|--------|-----------|:----------------:|
| **Unit** | Funções isoladas (NL parser, SQL builder, glossário) | pytest, Vitest | > 80% |
| **Integration** | API endpoints, banco, LLM mock | pytest + testcontainers | > 60% |
| **E2E** | Fluxo completo: signup → onboarding → query → gráfico | Playwright | Smoke (5 cenários críticos) |
| **Load** | Queries concorrentes por tenant, limites de plano | Locust | 50 queries simultâneas |
| **Security** | OWASP Top 10, SQL injection no NL engine, tenant isolation | OWASP ZAP + custom | Antes de cada release |
