### 2.3 Jornada As-Is (Como resolvem hoje)

```
CENÁRIO: Ana precisa saber "faturamento por região no último trimestre"

1. Ana identifica a necessidade durante uma reunião
   ↓
2. Ana abre ticket no sistema de chamados do time de dados
   ↓ (espera 2-8 horas para triagem)
3. Analista de dados prioriza entre 15-30 tickets na fila
   ↓ (espera 1-3 dias para execução)
4. Analista de dados escreve SQL, executa, valida resultado
   ↓ (30 min - 2 horas de trabalho técnico)
5. Analista de dados formata resultado em planilha/email e envia para Ana
   ↓
6. Ana recebe a tabela bruta — sem gráfico, sem insight, sem contexto
   ↓
7. Ana reformata em Excel para apresentar em reunião
   ↓ (30 min - 1 hora)
8. Na reunião, surge nova pergunta ("e por produto?") → volta ao passo 2

TOTAL: 2-5 dias para uma pergunta simples
CUSTO OCULTO: Tempo do analista de dados (~2h) + tempo da Ana (~1.5h) + decisão atrasada
```

### 2.4 Jornada To-Be (Com Veezoozin)

```
CENÁRIO: Ana precisa saber "faturamento por região no último trimestre"

1. Ana abre o Veezoozin no browser
   ↓
2. Ana digita em PT-BR: "Qual foi o faturamento por região no último trimestre?"
   ↓ (< 1 segundo)
3. Veezoozin analisa o contexto do tenant (glossário, schema, histórico)
   ↓ (< 2 segundos)
4. Veezoozin gera SQL contextualizado e executa no BigQuery (read-only, sandbox)
   ↓ (< 3 segundos)
5. Veezoozin retorna:
   - Gráfico de barras por região (tipo escolhido automaticamente)
   - Insight: "Região Sul cresceu 15% vs trimestre anterior"
   - Sugestão: "Quer ver o breakdown por produto?"
   ↓ (< 5 segundos total)
6. Ana clica na sugestão → nova query em < 5 segundos
   ↓
7. Ana exporta relatório visual para apresentar na reunião

TOTAL: < 30 segundos para a mesma pergunta
ITERAÇÃO: Instantânea — nova pergunta = nova resposta em segundos
```

### Qualidade das personas

**Avaliação: BOA** — O briefing define 4 personas com clareza de função e frequência de uso. Não é genérico ("para todo mundo"). A segmentação por papel (usuário final vs admin vs TI) é adequada para um SaaS B2B.

| 1 | **Falta persona de "Champion"** — quem dentro da empresa cliente vai evangelizar o produto? Não é o gestor (decisor) nem o analista (usuário). É alguém que "descobre" o Veezoozin e convence internamente. | Média | Sem champion identificado, a estratégia de go-to-market fica mais difícil |
| 2 | **Falta persona negativa** — quem NÃO deveria usar o Veezoozin? Data engineers? DBAs? Definir exclusões ajuda a focar o produto | Baixa | Risco de feature requests de personas erradas |
| 3 | **Objeções não mapeadas** — "E se a query estiver errada?" é objeção provável de Ana e Eduardo. O briefing não aborda confiança/validação de queries | Alta | Blocker de adoção se não endereçado |
| 4 | **Jornada do admin de tenant pouco detalhada** — O onboarding (glossário, schema mapping) é o momento mais crítico para time-to-value. Se for complexo, Daniel desiste | Alta | Time-to-value ruim = churn alto |

### Análise da jornada

**Jornada As-Is:** Bem mapeada. O ciclo "pergunta → ticket → espera → tabela bruta → reformatar → nova pergunta" é real e doloroso. A estimativa de 2-5 dias é conservadora — em empresas maiores, pode ser 1-2 semanas.

**Jornada To-Be:** Convincente, mas otimista. A latência < 5 segundos para "queries simples" é factível com BigQuery + LLM API, mas precisa de definição:
- O que é "query simples"? (1 tabela, sem joins complexos, sem window functions)
- O que acontece com queries complexas? (multi-join, subqueries, dados de períodos longos)
- Taxa de precisão declarada (> 85%) significa que 15% das queries estarão erradas — como o usuário detecta?

### Consistência com Bloco 1.1

- **Problema ↔ Personas:** Consistente. As 4 personas mapeiam diretamente para as 4 dores do problema declarado.
- **Diferenciação ↔ Jornada to-be:** Consistente. Multi-idioma, contexto por tenant e output visual aparecem na jornada.
- **Gap:** O bloco 1.1 recomendou cortar MCP/RAG do MVP. Se cortado, a persona do Admin de Tenant perde parte do valor (integração de conhecimento externo). Impacto aceitável — o glossário de negócio sozinho já entrega valor.

## 3. Recomendações do PO

### R1 — Mapear persona "Champion" (influenciador interno)

**Proposta:** Adicionar persona "Líder de equipe de BI/Dados" — profissional que conhece as dores do time E tem influência para propor novas ferramentas. É quem descobre o Veezoozin, faz o trial e convence a diretoria.

**Justificativa:** Em SaaS B2B com ticket < R$1.500, a venda é bottom-up (PLG/champion-led), não top-down. Sem champion identificado, a estratégia de aquisição fica cega.

**Ação:** Fabio valida com 2-3 líderes de BI se a dor descrita ressoa.
**Responsável:** Fabio
**Prazo:** Semana 2 do MVP
**Custo:** Zero (conversas informais)
**Consequência se não resolver:** Go-to-market sem canal de aquisição definido

### R2 — Endereçar objeção de confiança na query gerada

**Proposta:** Incluir no MVP funcionalidade de "transparência da query":
1. Mostrar o SQL gerado ao lado do resultado (toggle "Ver SQL")
2. Indicador de confiança: "Alta / Média / Baixa" baseado na complexidade da query e match com o glossário
3. Botão "Essa resposta está errada" para feedback que alimenta o aprendizado

**Justificativa:** Com taxa de precisão de 85%, 1 em cada 7 queries terá erro. Se o usuário não consegue detectar o erro NEM dar feedback, a confiança no produto degrada rapidamente. Transparência é table stakes para NL-to-SQL.

**Classificação:** Mandatório para MVP
**Responsável:** Fabio (desenvolvimento)
**Prazo:** Incluir no escopo MVP
**Custo:** ~1-2 sprints de desenvolvimento
**Consequência se não resolver:** Usuários perdem confiança após primeiros erros → churn no mês 2

### R3 — Priorizar onboarding do Admin de Tenant como feature crítica

**Proposta:** O fluxo de onboarding de tenant (setup do glossário + schema mapping) deve ser a segunda feature mais polida do MVP, depois da própria consulta NL-to-SQL. Proposta de fluxo:
1. Admin conecta BigQuery (credencial service account)
2. Sistema faz scan automático do schema (tabelas, colunas, tipos)
3. Sistema sugere descrições semânticas para colunas baseado em nome e dados sample
4. Admin confirma/corrige as sugestões
5. Sistema já está pronto para receber perguntas

**Justificativa:** Time-to-value para o admin é o momento mais frágil. Se levar mais de 30 minutos para configurar, o admin desiste e o trial morre. Auto-discovery de schema com IA reduz atrito drasticamente.

**Classificação:** Mandatório para MVP
**Responsável:** Fabio
**Prazo:** Sprint 2-3 do MVP
**Custo:** ~2 sprints de desenvolvimento
**Consequência se não resolver:** Onboarding complexo → time-to-value alto → trial não converte

### R4 — Definir persona negativa explicitamente

**Proposta:** Documentar que o Veezoozin NÃO é para:
- Data engineers (que já sabem SQL e preferem controle total)
- Cientistas de dados (que precisam de datasets completos, não de consultas pontuais)
- DBAs (que gerenciam infraestrutura, não consomem dados)

**Justificativa:** Persona negativa protege o roadmap de feature requests que não agregam valor ao público principal. Se um data engineer pede "acesso ao BigQuery console integrado", a resposta é "não é nossa persona".

**Classificação:** Desejável — documentar no product brief
**Responsável:** Fabio
**Prazo:** Antes do launch
**Custo:** Zero
**Consequência se não resolver:** Roadmap poluído com features de personas erradas
