**Trade-off Cloud SQL vs Firestore para metadados:**

| Critério | Cloud SQL | Firestore |
|----------|-----------|-----------|
| Custo fixo | ~R$250/mês (instância mínima) | R$0 (pay-per-op) |
| Relações | Nativo (FK, joins) | Manual (denormalização) |
| Queries complexas | SQL nativo | Limitado (composto queries) |
| Schema enforcement | Strong (migrations) | Flexible (schemaless) |
| **Recomendação** | **Usar para tenant/user/schema/billing metadata** | **Usar para sessões/cache/histórico** |

**Recomendação do Arquiteto:** Manter ambos. Cloud SQL para dados estruturados com relações (tenants, users, schemas, glossários, billing). Firestore para dados semi-estruturados de alto volume (sessões, cache, histórico). O custo de R$250/mês é trivial vs o benefício de ter relações nativas para metadata complexo.

## 2. Análise do Especialista

### Avaliação da stack

**Veredicto: ADEQUADA para o escopo e restrições.**

A stack proposta (Next.js + FastAPI + LangChain + GCP serverless) é moderna, bem documentada, e alinhada com as skills do time. Não há tecnologia exótica nem aposta arriscada.

**Trade-offs identificados:**

| Decisão | Prós | Contras | Veredicto |
|---------|------|---------|-----------|
| Next.js 14 (App Router) | SSR, RSC, SEO, TypeScript nativo | Complexidade do App Router, hydration issues | Aceitar — benefícios superam para SaaS |
| FastAPI (Python) | Async nativo, tipagem, OpenAPI auto, ecossistema ML | Python mais lento que Go/Rust para APIs de alta throughput | Aceitar — throughput não é bottleneck (BigQuery é) |
| LangChain | Abstração poderosa, chains, prompts, tools | Abstração pesada, debug difícil, breaking changes frequentes | Aceitar com ressalva — monitorar se LangChain vira gargalo |
| Cloud SQL (PostgreSQL) | Relações, SQL, migrations, maturidade | Custo fixo R$250/mês (único custo fixo de infra) | Aceitar — custo trivial, benefício alto |
| Firestore | Serverless, pay-per-op, flexible schema | Queries limitadas, sem joins, vendor lock-in | Aceitar — ideal para sessões e cache |

### Riscos técnicos

| # | Risco | Probabilidade | Impacto | Mitigação |
|---|-------|-------------|---------|-----------|
| 1 | **Prompt injection via NL-to-SQL** — Usuário manipula o LLM para gerar SQL malicioso | Alta | Crítico | Validação AST pós-LLM, whitelist de tabelas, reject non-SELECT |
| 2 | **LangChain breaking changes** — LangChain tem releases frequentes com breaking changes | Média | Médio | Pinnar versão, atualizar mensalmente com testes de regressão |
| 3 | **Latência > 5s para queries complexas** — LLM API (1-3s) + BigQuery (1-5s) pode exceder target | Alta | Médio | Cache de queries similares em Firestore, otimizar prompts |
| 4 | **Custo inesperado de BigQuery** — Query mal formada processa TB de dados | Média | Alto | Budget alert por tenant, limite de bytes por query (1GB) |
| 5 | **API key leakage (BYOK)** — API key do tenant vaza via logs ou error messages | Baixa | Crítico | Secret Manager, never log keys, redact em error responses |

### Mitigação detalhada — Risco 1 (Prompt Injection)

| Campo | Detalhe |
|-------|---------|
| **Ação concreta** | Implementar 3 camadas de defesa: (1) System prompt com instrução "gere APENAS SELECT statements"; (2) Parse da query gerada com sqlparse para validar que é SELECT; (3) Verificar tabelas referenciadas contra whitelist do schema do tenant |
| **Responsável** | Fabio |
| **Custo estimado** | ~3-5 dias de desenvolvimento (sprint 3-4) |
| **Prazo** | Sprint 3 (junto com o NL-to-SQL engine) |
| **Consequência se não resolver** | Usuário mal-intencionado pode extrair dados de tabelas fora do escopo ou causar danos no BigQuery (mesmo com read-only, queries caras são possíveis) |

### Mitigação detalhada — Risco 5 (API Key Leakage)

| Campo | Detalhe |
|-------|---------|
| **Ação concreta** | (1) Armazenar API keys no Secret Manager (nunca em banco); (2) Em runtime, carregar keys em memória efêmera; (3) Middleware de logging com redação automática de patterns de API key; (4) Audit log de acesso a keys; (5) Permitir que tenant revogue e rotacione keys pelo painel |
| **Responsável** | Fabio |
| **Custo estimado** | ~2-3 dias de desenvolvimento (sprint 2) |
| **Prazo** | Sprint 2 (junto com BYOK) |
| **Consequência se não resolver** | API key vazada = tenant tem custos não autorizados em seu provedor de LLM. Veezoozin perde credibilidade e potencialmente tem liability legal. |

## 3. Recomendações do Arquiteto

### R1 — Usar Firebase Auth como provedor de autenticação

**Proposta:** Firebase Authentication para auth no MVP. Suporta email/senha, Google Sign-In, e é managed service GCP. Migrar para Identity Platform (mesma API, mais features) quando precisar de SAML/OIDC para Enterprise.

**Justificativa:** Elimina 2-3 sprints de código custom de auth. Integra nativamente com Cloud Run e Firestore. Upgrade path claro para SSO enterprise.

**Alternativa avaliada:** Auth.js (NextAuth) — bom, mas requer mais código custom e não integra nativamente com GCP.

### R2 — Implementar validação de SQL em 3 camadas

**Proposta:** Toda query gerada pelo LLM passa por 3 gates antes de execução:
1. **Gate 1 — LLM System Prompt:** Instrução explícita "gere apenas SELECT, sem DDL/DML"
2. **Gate 2 — SQL Parser:** Parse com `sqlparse` para confirmar statement type = SELECT
3. **Gate 3 — Schema Validator:** Verificar que todas as tabelas/colunas referenciadas existem no schema mapeado do tenant

**Justificativa:** Prompt injection é o risco #1 de segurança em NL-to-SQL. Defesa em profundidade é obrigatória.

### R3 — Implementar budget control por tenant no BigQuery

**Proposta:** Cada tenant tem um budget diário de bytes processados no BigQuery (ex: 10GB/dia para Starter, 50GB/dia para Pro, 200GB/dia para Enterprise). Query que excede o budget é rejeitada com mensagem explicativa.

**Justificativa:** BigQuery cobra $6.25/TB processado. Uma query sem filtro em tabela grande pode custar R$50+ em uma única execução. Sem budget control, custo variável por tenant pode explodir.

### R4 — Avaliar alternativa ao LangChain (decision point na semana 4)

**Proposta:** Iniciar com LangChain mas fazer PoC paralela com chamada direta à API do LLM (Claude/Gemini) com prompt templates custom. Na semana 4, decidir: se LangChain adiciona valor claro (chains complexos, tool use), manter. Se é apenas wrapper sem benefício, dropar em favor de chamadas diretas.

**Justificativa:** LangChain é o framework mais popular, mas muitos projetos NL-to-SQL descobrem que chamada direta com prompt engineering é mais simples e previsível. Decision point cedo evita sunk cost.

### R5 — Secret Manager para TODAS as credenciais sensíveis

**Proposta:** Zero credenciais em environment variables do Cloud Run (exceto referências ao Secret Manager). Todas as API keys (LLM BYOK, BigQuery service accounts, Stripe) no Secret Manager com rotação automática.

**Justificativa:** Environment variables em Cloud Run são visíveis no console GCP para qualquer pessoa com acesso ao projeto. Secret Manager tem controle de acesso granular e audit log.
