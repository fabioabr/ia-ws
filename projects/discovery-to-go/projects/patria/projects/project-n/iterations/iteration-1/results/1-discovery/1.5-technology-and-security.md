# Technology and Security — FinTrack Pro

> **Fase:** 1 — Discovery | **Iteracao:** 1 | **Bloco:** 1.5 | **Status:** Rascunho

## Stack Tecnologica

| Camada | Tecnologia | Justificativa |
|--------|-----------|---------------|
| Backend | Node.js (TypeScript) | Equipe com experiencia comprovada; ecossistema rico para integracao com APIs financeiras |
| Frontend | React | Biblioteca madura para dashboards interativos; conhecimento existente na equipe |
| Banco de dados | PostgreSQL | Suporte robusto a transacoes ACID, JSON nativo para dados semi-estruturados de extratos bancarios |
| Fila assincrona | BullMQ (Redis) | Processamento de ingestao bancaria em background sem bloquear a aplicacao principal |
| CI/CD | GitHub Actions | Pipeline ja existente na empresa, sem custo adicional de adocao |

A escolha da stack prioriza **produtividade da equipe atual** (5 pessoas) sobre otimizacao teorica. Toda a equipe — CTO full-stack senior, 2 devs backend pleno, 1 frontend pleno — ja domina Node.js/TS + React + PostgreSQL.

## Integracao Open Finance

| Criterio | Belvo | Pluggy |
|----------|-------|--------|
| Cobertura de bancos BR | Alta | Alta |
| API para Open Finance Brasil | Sim | Sim |
| Suporte a consentimento nativo | Sim | Sim |
| Documentacao e SDK Node.js | Boa | Boa |
| Custo inicial (MVP) | Medio | Medio |

**Decisao:** Pluggy selecionada apos Human Review da iteracao 1. Ambos os providers atendem aos requisitos tecnicos; a decisao considerou facilidade de integracao com o fluxo de consentimento do Open Finance e suporte local.

> **Pendencia resolvida:** A escolha entre Belvo e Pluggy estava pendente durante a entrevista e foi definida na etapa de revisao.

## Estrategia de IA

| Aspecto | Decisao |
|---------|---------|
| Abordagem MVP | Consumo de LLM via API externa (OpenAI / Claude) |
| Modelos proprios | Nao — fora do escopo do MVP |
| Casos de uso | Projecoes de fluxo de caixa e deteccao de anomalias em gastos |
| Escala futura | Avaliacao de modelos proprietarios na Fase 3 |

A estrategia evita o custo e complexidade de treinar modelos proprios com uma equipe de 5 pessoas. APIs externas permitem iterar rapidamente sobre os prompts e ajustar a qualidade das projecoes sem investimento em infraestrutura de ML.

## Autenticacao e Acesso

| Recurso | Disponibilidade | Detalhe |
|---------|----------------|---------|
| MFA (Multi-Factor Authentication) | Todos os planos | Obrigatorio — requisito de seguranca para dados financeiros sensiveis |
| SSO (SAML 2.0) | Somente Enterprise | Integracao com identity providers corporativos (Azure AD, Okta) |
| Sessao | Todos os planos | Token JWT com expiracao curta + refresh token |

MFA obrigatorio desde o plano Starter reflete a natureza sensivel dos dados financeiros tratados pelo sistema.

## Cloud e Infraestrutura

| Aspecto | Decisao |
|---------|---------|
| Provider | AWS |
| Justificativa | Conta existente com creditos do programa AWS Activate para startups |
| Servicos principais | ECS/Fargate (compute), RDS PostgreSQL, ElastiCache Redis, S3 (armazenamento), CloudWatch (monitoramento) |
| Ambiente MVP | Single-region (sa-east-1 — Sao Paulo) |
| Escalabilidade | Horizontal via auto-scaling de containers; migracao para microservicos prevista na Fase 2 |

A escolha da AWS nao e apenas por creditos — a maturidade dos servicos gerenciados (RDS, ElastiCache) reduz o overhead operacional para uma equipe enxuta.
